# AISDLC 執行檢查清單

## 🎯 確保正確執行 AISDLC 框架

### 📋 執行前檢查（Before Execution）

#### ✅ 初始化檢查
- [ ] **載入配置文件**：已執行 `載入 AISDLC_INIT.md`
- [ ] **確認映射表**：LLM 已回應載入 workflow-agent 映射關係
- [ ] **選定 Workflow**：明確要執行的 workflow 名稱
- [ ] **準備材料**：已準備好輸入材料（截圖、文件、需求描述等）

#### ✅ Token 效率檢查
- [ ] **按需載入**：LLM 應採用按需載入，不是全量載入
- [ ] **輕量初始化**：初始化 token 消耗 < 300
- [ ] **智能緩存**：同一對話中重複使用的 agent 不重複載入

### 🔄 執行中檢查（During Execution）

#### ✅ Agent 載入驗證
- [ ] **載入顯示**：LLM 顯示「🔄 正在載入相關配置...」
- [ ] **Primary Agent**：正確載入主要負責的 Agent
- [ ] **Supporting Agents**：載入所有支援協作的 Agents  
- [ ] **配置確認**：顯示「✅ 套用所有 Agent 規則」

#### ✅ 規則執行檢查
- [ ] **Agent 人格**：執行風格符合對應 Agent 的人格設定
- [ ] **Core Principles**：遵循 Agent 的核心原則
- [ ] **Collaboration Rules**：Supporting Agents 在對應環節參與
- [ ] **Quality Standards**：產出符合 Agent 的品質標準

#### ✅ 人機協作驗證
- [ ] **強制確認點**：在所有 🔴 標記處暫停等待確認
- [ ] **不確定就問**：遇到模糊情況主動詢問用戶
- [ ] **標準選項**：提供編號選項供用戶選擇
- [ ] **影響分析**：說明每個選項的影響和後果

#### ✅ Workflow 步驟檢查
- [ ] **步驟順序**：按照 workflow 定義的順序執行
- [ ] **必要步驟**：不跳過任何 MANDATORY 步驟
- [ ] **輸入驗證**：驗證每個步驟的輸入是否完整
- [ ] **輸出檢查**：每個步驟產生預期的輸出

### 📊 執行後檢查（After Execution）

#### ✅ 文檔產出驗證
- [ ] **模板符合度**：產出文檔符合 AISDLC 模板格式
- [ ] **內容完整性**：包含所有必要的章節和內容
- [ ] **追蹤鏈完整**：建立從需求到實現的完整追蹤鏈
- [ ] **格式標準化**：使用標準的編號格式（US-XXX, AC-XXX-X 等）

#### ✅ 品質標準檢查
- [ ] **需求理解準確**：正確理解和映射用戶需求
- [ ] **業務邏輯合理**：業務流程和規則設計合理
- [ ] **技術可行性**：技術方案可行且合理
- [ ] **驗收標準可測**：驗收標準具體可測試

#### ✅ 協作記錄檢查  
- [ ] **決策記錄**：記錄所有重要決策過程
- [ ] **確認點記錄**：記錄每個確認點的選擇和理由
- [ ] **變更追蹤**：如有變更，記錄變更歷史
- [ ] **經驗總結**：總結執行過程中的經驗和建議

## 🚨 常見問題診斷

### 問題類型 A：載入相關問題

#### A1. LLM 沒有按需載入
**症狀**：直接執行 workflow，沒有載入過程
**診斷**：
- [ ] 檢查是否載入了 AISDLC_INIT.md
- [ ] 確認 LLM 是否理解按需載入機制
**修復**：重新提醒載入配置文件

#### A2. Agent 文件讀取失敗  
**症狀**：顯示「無法載入 [agent].yaml」
**診斷**：
- [ ] 檢查 agent/ 目錄是否存在
- [ ] 確認對應 agent 文件是否完整
- [ ] 檢查文件路徑是否正確
**修復**：確保文件結構完整

#### A3. Token 消耗過多
**症狀**：初始化消耗 > 500 tokens
**診斷**：
- [ ] 檢查是否意外載入了全量配置  
- [ ] 確認是否重複載入相同 agent
**修復**：重新執行按需載入

### 問題類型 B：執行相關問題

#### B1. 跳過確認點
**症狀**：LLM 沒有在 🔴 處暫停
**診斷**：
- [ ] 檢查 Agent 規則是否正確載入
- [ ] 確認 workflow 的確認點設定
**修復**：強調必須執行所有 MANDATORY 確認點

#### B2. Agent 人格不符
**症狀**：執行風格不符合預期的 Agent 特色
**診斷**：
- [ ] 檢查是否載入了正確的 Agent  
- [ ] 確認 Agent 的 core_principles 是否被套用
**修復**：要求重新載入 Agent 配置

#### B3. 協作 Agent 缺席
**症狀**：Supporting Agents 沒有參與對應環節
**診斷**：
- [ ] 檢查映射表中的 Supporting Agents 設定
- [ ] 確認 collaboration_rules 是否執行
**修復**：明確要求載入並執行協作 Agent

### 問題類型 C：產出相關問題

#### C1. 文檔格式不符
**症狀**：產出不符合 AISDLC 模板
**診斷**：
- [ ] 檢查是否使用了正確的文檔模板
- [ ] 確認 Agent 的品質標準是否執行
**修復**：要求按照指定模板重新產出

#### C2. 內容不完整
**症狀**：缺少必要的章節或內容
**診斷**：
- [ ] 檢查 workflow 是否完整執行
- [ ] 確認是否跳過了某些步驟
**修復**：補充缺失的內容和步驟

## 🎯 執行效果評估

#### Token 效率
- **優秀**：< 600 tokens (節省 > 70%)
- **良好**：600-900 tokens (節省 50-70%)  
- **及格**：900-1200 tokens (節省 30-50%)
- **需改進**：> 1200 tokens

#### 執行時間
- **單一 Workflow**：< 60 分鐘
- **組合 Workflow**：< 120 分鐘
- **完整流程**：< 180 分鐘

#### 文檔品質
- **格式正確率**：> 95%
- **內容完整率**：> 90%  
- **可執行性**：> 85%

## 🔄 持續改進建議

### 基於檢查結果的行動

#### 如果執行效果優秀
- 記錄成功案例
- 分享最佳實踐
- 考慮擴展到更多場景

#### 如果執行效果良好  
- 分析可改進的環節
- 調整 Agent 配置或 Workflow 步驟
- 提升確認點的有效性

#### 如果執行效果不佳
- 檢查框架配置是否正確
- 重新訓練 LLM 的執行模式
- 考慮簡化複雜的流程

### 框架優化方向
1. **提高 Token 效率**：進一步優化載入機制
2. **改進用戶體驗**：減少不必要的確認點
3. **增強穩定性**：加強錯誤處理和恢復機制
4. **擴展適用性**：支持更多場景和使用方式
