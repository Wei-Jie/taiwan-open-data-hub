# AISDLC 框架使用手冊

> **版本**：v1.0 | **最後更新**：2026-05-03  
> **適用對象**：所有使用 AISDLC 框架的開發者與 AI 協作者

---

## 第一章：框架概覽

### 1.1 什麼是 AISDLC？

AISDLC（AI-Assisted Software Development Life Cycle，AI 輔助軟體開發生命週期）是一個結構化框架，利用大型語言模型（LLM）與 AI 代理人（Agents）輔助軟體開發各個階段。

**核心理念**：
- 🛡️ **反幻覺優先**：「寧可空白，不可捏造」
- 🤝 **人機協作**：關鍵決策點必須等待人類確認
- 📄 **文件驅動**：所有開發圍繞標準化文件進行
- ⚡ **按需載入**：只載入當前需要的 Agent，節省 74% Token

### 1.2 三大核心支柱

| 支柱 | 說明 | 資料夾 |
|------|------|--------|
| **Agents（代理人）** | 模擬不同角色的 AI 人格 | `agent/` |
| **Workflows（工作流程）** | 可重複使用的標準化流程 | `workflow/` |
| **Docs（文件）** | 開發過程的核心產出 | `docs_template/` |

## 第二章：快速開始

### 2.1 整合框架（3 步驟）

**Step 1：加入專案**
```bash
# 推薦方式：Git Submodule
cd your-project/
git submodule add <AISDLC-repo-url> aisdlc
git submodule update --init

# 替代方式：直接複製
# 將整個 AISDLC 資料夾複製到你的專案根目錄
```

**Step 2：載入初始化配置**  
在任何 LLM 工具（ChatGPT / Claude / Copilot / Gemini 等）中輸入：
```
請載入 aisdlc/AISDLC_INIT.md 文件，建立按需載入的 workflow-agent 映射關係。
```

**Step 3：確認載入成功**  
LLM 應回應類似：
```
✅ AISDLC 按需載入機制已啟用！
🛡️ 反幻覺規則已強制啟用（最高優先級）
映射表已載入：9個 Workflow → 對應 Agent
準備接收 workflow 指令！
```

### 2.2 第一次使用（30 秒開始）

複製以下模板，貼入你的 AI 對話中：
```
我需要分析需求，材料如下：

[在這裡貼上你的截圖描述或文字需求]

請執行完整的 AISDLC 需求分析流程，生成所有開發文檔。
過程中有疑問請詢問我確認。
```

## 第三章：Workflow 工作流程

### 3.1 快速選擇表

| 你有什麼？ | 你想得到什麼？ | 使用 Workflow |
|----------|-------------|--------------|
| 截圖 / 想法 | 需求分析報告 | 1️⃣ 需求提取 |
| 需求報告 | 正式 PRD/FRD | 2️⃣ 需求驗證與文檔化 |
| PRD/FRD | 開發規格 | 3️⃣ 用戶故事與系統設計 |
| 現有需求 | 修改需求 | 4️⃣ 需求變更管理 |
| SRD 文檔 | API 規格 | 5️⃣ API 規格生成 |
| 專案文檔 | 品質檢查 | 6️⃣ 文檔一致性檢查 |
| SRD 缺交互流程 | 前後端設計 | 7️⃣ 前後端交互分析 |
| 要 TDD 開發功能 | 測試驅動實作 | 8️⃣ TDD 開發流程 |

### 3.2 觸發 Workflow 的方式

每個 Workflow 對應一個 Prompt 模板，位於 `prompts/workflow-prompts/` 目錄：

```
執行 requirements-extraction workflow 分析我的需求：
[貼上你的截圖或需求描述]
```

### 3.3 九大 Workflow 詳解

#### 1️⃣ 統一需求提取（unified-requirements-extraction）
- **適用**：有截圖或文字想法，需要結構化分析
- **輸入**：截圖、文字、或兩者混合
- **輸出**：需求提取報告、人機協作記錄
- **主要 Agent**：SA（Amanda）+ BA（Beatrice）
- **特色**：4個強制人機確認點

#### 2️⃣ 需求驗證與文檔化（validation-documentation）
- **適用**：需求已提取，要驗證並生成正式文件
- **輸入**：需求提取報告
- **輸出**：PRD（產品需求文件）、FRD（功能需求文件）
- **主要 Agent**：SA（Amanda）、BA（Beatrice）、PM/PO（Victoria）

#### 3️⃣ 用戶故事與系統設計（user-story-design）
- **適用**：有 PRD/FRD，要轉換為開發規格
- **輸入**：PRD、FRD 文件
- **輸出**：User Stories、SRD（系統需求文件）、開發計畫
- **主要 Agent**：SA（Amanda）+ SD（Marcus）

#### 4️⃣ 需求變更管理（change-management）
- **適用**：現有功能需要修改
- **輸入**：現有文件 + 變更描述
- **輸出**：變更分析報告、更新的文件
- **主要 Agent**：SA、BA、PM/PO、SD、QA 協作
- **特色**：6個人機協作確認點

#### 5️⃣ API 規格生成（api-specification）
- **適用**：需要生成或更新 API 規格
- **輸入**：SRD 文件、User Story
- **輸出**：每個 API 的獨立規格文件（`API_[模組]_[端點].md`）
- **主要 Agent**：SD（Marcus）+ QA（Quincy）

#### 6️⃣ 文檔一致性檢查（consistency-check）
- **適用**：檢查專案文件品質
- **輸入**：所有專案文件（PRD、FRD、SRD、AT 等）
- **輸出**：一致性檢查報告、修復建議
- **主要 Agent**：SA + SD + QA

#### 7️⃣ 前後端交互分析（interaction-analysis）
- **適用**：SRD 缺少交互流程描述
- **輸入**：現有 SRD、FRD
- **輸出**：Mermaid 交互序列圖、更新的 SRD
- **主要 Agent**：SD（Marcus）+ Dev（David）

#### 8️⃣ TDD 功能開發流程（tdd-development）
- **適用**：單一功能的測試驅動開發
- **輸入**：User Story、Acceptance Criteria、SRD
- **輸出**：測試代碼、實作代碼、測試報告
- **主要 Agent**：Dev（David）+ SA + SD + QA
- **特色**：紅燈→綠燈→重構完整 TDD 循環

#### 9️⃣ 需求追蹤分析（requirement-traceability）
- **適用**：口語化需求需要對應到正式文件
- **輸入**：用戶需求描述、專案文件
- **輸出**：需求追蹤矩陣
- **說明**：通常由 TDD Workflow 自動觸發，無需手動執行

## 第五章：反幻覺規則

### 5.1 核心原則（強制遵守）

> 🛡️ **最高優先級：寧可空白，不可捏造**

1. **事實檢查**：回答前必須進行事實檢查，禁止假設、推測或自行創造內容
2. **嚴格依據來源**：僅使用使用者明確提供或可驗證的資料
3. **零推測原則**：遇到任何需要假設的情況，必須立即停止並詢問人類
4. **顯示依據**：所有推論必須說明來源
5. **誠實聲明不確定**：資訊不足時直接說「沒有足夠資料」
6. **保持語意一致**：不可改寫或擴大使用者原意

### 5.2 禁止出現的語句

以下模糊語句一律禁用：
- ❌「應該是」、「可能是」、「我猜」
- ❌「通常來說」、「一般情況下」
- ❌「按照慣例」、「根據經驗」、「行業標準做法」

### 5.3 執行前快速自檢

提交任何重要輸出前，確認：
1. 我的每一項陳述都有明確來源嗎？
2. 我是否做了任何未經確認的假設？
3. 我是否使用了任何模糊語氣？
4. 遇到不確定時我是否詢問了人類？
5. 我的輸出是否可以完全追溯到輸入？

## 第七章：安全性強化（Project Hardening）

### 7.1 觸發時機

以下情況應啟動 `SKILL_Project_Hardening.md`：
1. 🆕 新專案啟動（建立專案目錄後）
2. 🏗️ 架構設計（撰寫 SRD 前）
3. 🔧 涉及 API、資料庫或身份驗證的重構

### 7.2 必須確認的安全項目

**併發保護**
- [ ] 是否有 Race Condition 風險？（如 GAS LockService）
- [ ] 流水號是否可能重疊？

**流量防禦**
- [ ] API 是否有限流（Rate Limit）？
- [ ] CORS 是否限制允許的 Domain？
- [ ] 後端是否對所有傳入參數進行型別與長度檢查？

**維運監控**
- [ ] 是否有全域錯誤攔截（Global Catch）？
- [ ] 是否有系統日誌？
- [ ] 是否使用動態 Header Mapping 防止欄位移位？

**環境安全**
- [ ] `.env` 是否已加入 `.gitignore`？
- [ ] Log 中是否會不經意記錄敏感資料？

## 第八章：常見問題排解

### Q1：LLM 沒有按需載入流程
**症狀**：直接執行 Workflow，沒有顯示載入過程  
**解決**：
```
提醒 LLM：「請按照 AISDLC_INIT.md 的按需載入流程，
先載入對應的 agent 配置再執行 workflow」
```

### Q2：LLM 跳過確認點
**症狀**：沒有在 🔴 標記處暫停  
**解決**：
```
提醒：「請嚴格執行所有 MANDATORY 確認點，
在每個 🔴 標記處等待我的確認」
```

### Q3：Agent 文件讀取失敗
**症狀**：顯示「無法載入 [agent].yaml」  
**解決**：
1. 確認 `agent/` 目錄是否完整
2. 若用 Submodule，確認已執行 `git submodule update --init`
3. 確保 LLM 有讀取文件的權限

### Q4：Agent 風格不符合預期
**症狀**：執行風格不符合 Agent 人格設定  
**解決**：
```
要求 LLM 重新載入：「請重新讀取 aisdlc/agent/sa-analyst.yaml
並嚴格按照其中的 core_principles 執行」
```

### Q5：文件格式不符合 AISDLC 標準
**症狀**：產出文件不符合模板格式  
**解決**：指定模板路徑要求重新產出
```
請按照 docs_template/frd/ 中的模板格式重新產出 FRD 文件
```

## 第九章：最佳實踐

### 9.1 DO（推薦做法）
- ✅ 每次對話開始前載入 `AISDLC_INIT.md`
- ✅ 讓 LLM 顯示載入過程，確認配置正確
- ✅ 耐心配合所有確認點，提供充分的回應
- ✅ 定期使用「文檔一致性檢查」Workflow 驗證品質
- ✅ 誠實表達不確定的地方，不要猜測

### 9.2 DON'T（避免做法）
- ❌ 跳過初始化直接使用 Workflow
- ❌ 催促 LLM 跳過確認點
- ❌ 修改核心 Workflow 文件（應 Fork 後修改）
- ❌ 在 LLM 資訊不足時強迫它繼續執行
- ❌ 忽略反幻覺規則的違規提示

### 9.3 Token 使用效率

|---------|--------|------------|------|--------|
| 傳統全載入 | ~2000 | +300 | ~2300 | — |
