# 前端專案 Rules 模板說明

## 概述

本文件說明如何使用前端專案 Rules 模板系統，為不同的前端專案建立一致的開發規範和工作流程。此模板系統分為兩大部分：**MCP Rules** 和 **Project Rules**，可根據專案需求進行客製化調整。

## 模板結構

```
ai-sdlc/docs_template/frontend/rules/
├── mcp-rules/                              # MCP 工具相關規則模板
│   ├── figma-mcp-template.mdc             # Figma MCP 讀取規則模板
│   └── playwright-ui-testing-template.mdc # Playwright UI 測試流程模板
└── project-rules/                          # 專案特定規則模板
    ├── 01-project-overview-template.mdc    # 專案概述模板
    ├── 02-components-template.mdc          # 組件系統模板
    ├── 03-authentication-template.mdc     # 身份驗證系統模板
    ├── 04-styling-template.mdc            # 樣式系統模板
    └── 05-data-handling-template.mdc      # 數據處理與 API 模板
```

## 第一部分：MCP Rules

MCP Rules 是與 Model Context Protocol 工具相關的規則，這些規則相對固定，適用於所有使用相應 MCP 工具的專案。

### 1. Figma MCP 讀取規則 (`figma-mcp-template.mdc`)

**用途**: 定義透過 Figma MCP 讀取設計稿的標準流程和格式

**主要內容**:

- 標準 Prompt 格式和查詢方式
- Frame 資訊分析要求（Frame ID、Frame Name、完整連結、互動關係）
- 標準回應格式規範
- 範圍控制和資料品質要求
- 品質保證檢查清單和驗證方法
- 工具使用說明（MCP 呼叫方式、參數解析）

**填寫說明重點**:

- 提供專案特定的命名格式範例
- 可調整 Frame 命名規範和存放路徑
- 包含完整的品質檢查流程

**使用時機**: 當專案需要從 Figma 讀取設計資料時

### 2. Playwright UI 測試流程 (`playwright-ui-testing-template.mdc`)

**用途**: 定義使用 Playwright MCP 執行自動化 UI 測試的標準流程

**主要內容**:

- 流程執行守則（10大核心原則）
- BDD 測試步驟序列化（Given/When/Then/Teardown）
- 數據結構標準（testId、meta、browser、steps）
- 待執行清單格式
- 結果記錄和報告格式（record_step、summarize_results）
- 錯誤處理機制和中斷處理

**填寫說明重點**:

- 測試案例ID和標題格式可客製化
- 瀏覽器配置可根據專案需求調整
- 步驟描述和預期結果格式可調整

**使用時機**: 當專案需要實施自動化 UI 測試時

## 第二部分：Project Rules

Project Rules 是與專案結構、技術棧和業務邏輯相關的規則，需要根據具體專案進行客製化。

### 1. 專案概述 (`01-project-overview-template.mdc`)

**用途**: 定義專案的整體架構、目錄結構和路由設計

**主要內容**:

- 專案基本資訊（名稱、描述、技術棧）
- 專案架構說明（前後台分離、多系統架構等）
- 目錄結構設計和功能特點
- 路由設計與導航機制（多層路由、域名重寫等）
- 開發流程規範和文件管理
- 技術棧說明和部署環境
- 修訂歷史記錄

**填寫說明重點**:

- 詳細的填寫指導和範例提供
- 支援單系統和多系統架構調整
- 包含完整的技術棧配置說明
- 提供修訂歷史管理格式

**涵蓋的大方向**:

- 專案架構設計原則
- 目錄組織規範
- 路由和導航機制
- 開發流程標準化

### 2. 組件系統 (`02-components-template.mdc`)

**用途**: 規範組件的組織、命名和使用方式

**主要內容**:

- UI 組件放置位置和命名規範
- 組件資料夾結構和組織原則
- 導航、表單、認證、功能性組件分類
- 媒體播放、圖標、表格組件管理
- 特定頁面組件和 Skeleton 組件規則
- 組件資料夾規則和最佳實踐

**填寫說明重點**:

- 適用於不同 UI 框架（React、Vue、Angular）
- 提供具體的命名格式和路徑範例
- 包含組件分類和組織的詳細指導
- 支援不同專案複雜度的組件管理

**涵蓋的大方向**:

- 組件分類和組織原則
- 命名規範和最佳實踐
- 可重用性和維護性考量
- 特定頁面組件管理

### 3. 身份驗證系統 (`03-authentication-template.mdc`)

**用途**: 定義身份驗證的架構和實作方式

**主要內容**:

- 核心認證文件和相關組件
- 路由保護和認證頁面組織
- 架構特點
- 頁面和組件中的使用方式
- 自定義 Hook 使用（針對 React）
- 環境配置和安全考量

**填寫說明重點**:

- 支援多種認證框架（NextAuth.js、Auth0、Firebase Auth等）
- 適用於不同前端框架的認證模式
- 包含單系統和多系統的認證架構
- 提供完整的安全配置指導

**涵蓋的大方向**:

- 認證架構設計
- 會話和狀態管理
- 安全性考量
- 用戶體驗優化

### 4. 樣式系統 (`04-styling-template.mdc`)

**用途**: 規範樣式的組織和管理方式

**主要內容**:

- 主要樣式文件和樣式模式
- 組件樣式和主題系統
- 響應式設計策略
- 樣式檔案結構和命名慣例
- 樣式最佳實踐
- 自定義樣式和組件專用樣式

**填寫說明重點**:

- 支援多種 CSS 框架（Tailwind CSS、Bootstrap等）
- 提供主題系統和響應式設計指導
- 包含樣式組織和命名規範
- 適用於不同的樣式架構模式

**涵蓋的大方向**:

- 樣式架構和組織
- 主題和設計系統
- 響應式設計原則
- 效能和維護性考量

### 5. 數據處理與 API (`05-data-handling-template.mdc`)

**用途**: 定義數據獲取、處理和 API 整合的規範

**主要內容**:

- 服務器操作和 API 路由組織
- 數據驗證和類型定義
- 環境變量和 API 版本配置
- API 界面設計（統一調用模式、Mock Data 模式）
- Coding Style 規範（TypeScript、API 函數、Server Actions）
- 表單提交邏輯規範和狀態管理
- API 安全考量

**填寫說明重點**:

- 適用於多種前端框架（Next.js、Vue.js、React等）
- 支援不同的狀態管理和驗證工具
- 提供完整的 Mock Data 開發模式
- 包含詳細的編碼規範和最佳實踐
- 強調規範的可調整性和靈活性

**涵蓋的大方向**:

- API 設計和整合模式
- 數據驗證和錯誤處理
- Mock Data 開發模式
- 表單提交和狀態管理

## 如何使用模板

### 1. 建立新專案的 Rules

1. **複製模板目錄**: 將 `ai-sdlc/docs_template/frontend/rules/` 複製到新專案的 `.cursor/rules/` 目錄
2. **重新命名文件**: 移除 `-template` 後綴，例如 `figma-mcp-template.mdc` → `figma-mcp.mdc`
3. **客製化 MCP Rules**: 根據專案實際使用的 MCP 工具調整相應規則
4. **客製化 Project Rules**: 根據專案的技術棧、架構和業務需求填入具體內容

### 2. 客製化指導原則

#### MCP Rules 客製化

- **保持核心邏輯不變**: MCP 工具的基本工作流程應該保持一致
- **調整專案特定內容**: 如 Frame 命名格式、截圖存放路徑、測試案例格式等
- **移除不需要的工具**: 如果專案不使用某個 MCP 工具，可以移除對應的規則文件

#### Project Rules 客製化

- **技術棧替換**: 將模板中的 `[技術框架]`、`[UI框架]` 等佔位符替換為實際使用的技術
- **結構調整**: 根據專案的實際目錄結構和組織方式調整模板內容
- **業務邏輯**: 加入專案特有的業務邏輯和架構設計
- **命名規範**: 統一專案的命名規範和編碼風格

### 3. 模板填空示例

以 `02-components-template.mdc` 為例：

**模板內容**:

```
本應用程序使用以下主要組件類型：

- 基礎 UI 組件位於 `[基礎組件目錄]` 目錄，基於 [UI框架] 。
- [認證組件目錄] - [認證相關組件集合]
  - `[登入組件]` - [登入組件功能]
```

**填入實際內容**:

```
本應用程序使用以下主要組件類型：

- 基礎 UI 組件位於 `@/components/ui` 目錄，基於 shadcn/ui 。
- `src/components/auth` - 身份驗證相關組件集合
  - `login-form.tsx` - 登入表單組件
```

### 4. 維護和更新

- **定期檢查**: 隨著專案發展，定期檢查 Rules 是否需要更新
- **版本控制**: 將 Rules 文件納入版本控制，確保團隊同步
- **文件同步**: 當程式碼架構變更時，同步更新對應的 Rules 文件
- **團隊協作**: 確保團隊成員都了解並遵循 Rules 規範

## 模板特色

### 詳細填寫指導

每個模板都包含豐富的 `_填寫說明：..._` 指導，提供：

- 具體的技術選項和範例
- 佔位符 `[變數名]` 的詳細說明
- 不同專案架構的適配建議
- 可選章節和移除條件說明

### 技術棧適應性

- **多框架支援**: 適用於 React、Vue、Angular、Next.js 等
- **工具選擇彈性**: 支援各種 UI 庫、狀態管理、認證方案
- **架構模式**: 涵蓋單系統、多系統、微前端等架構
- **部署環境**: 適應不同的開發和部署需求

### 規範靈活性

- **建議而非強制**: 提供最佳實踐建議，但強調可根據團隊需求調整
- **模組化設計**: 可以選擇性採用部分規則
- **擴展性**: 易於根據專案發展進行擴展和修改

## 最佳實踐

1. **漸進式採用**: 不需要一次性實作所有 Rules，可以根據專案需要逐步引入
2. **團隊共識**: 在採用 Rules 前，確保團隊成員達成共識
3. **持續改進**: 根據實際使用經驗，持續優化和改進 Rules 內容
4. **工具整合**: 將 Rules 與 linting 工具、CI/CD 流程整合，確保自動化執行
5. **文件化**: 保持 Rules 文件的清晰和完整，便於新成員理解和遵循
6. **定期回顧**: 定期檢視和更新 Rules，確保與專案發展同步

## 注意事項

1. **規範彈性**: 這些模板提供的是開發規範的框架，具體的編碼風格和實作細節應根據專案實際需求調整
2. **技術適應**: 填寫時應選擇適合專案的技術棧，不必強制使用模板中的所有選項
3. **團隊協作**: 規範的制定和執行需要整個開發團隊的參與和共識
4. **持續演進**: Rules 應該隨著專案發展和技術更新而持續演進

---

透過使用這套模板系統，可以確保不同前端專案在開發規範、工作流程和程式碼品質方面保持一致性，同時又具備足夠的靈活性來適應各種技術棧和業務需求。每個模板都包含詳細的填寫指導，讓團隊能夠快速建立符合專案特色的開發規範。
