# 開發安全規範：使用者帳號與密碼原則實作

**版本**: 3.0
**日期**: 2025-09-04
**適用範圍**: 身份認證與帳號管理模組的程式碼實作。

## 1. 核心認證策略 (Core Authentication Strategy)

### CG-SEC-000: 【強制】優先採用 SSO 整合

- **規範**:
  1.  **優先級**: 系統**必須**優先評估並整合公司統一的單一登入 (SSO) 系統。
  2.  **責任轉移**: 一旦確認採用 SSO 方案，本應用程式**不應**自行儲存或管理使用者密碼。所有密碼相關的安全原則（包括密碼複雜度、變更週期、登入失敗鎖定）將由 SSO 身份提供者 (IdP) 統一負責。
  3.  **合規性認定**: 成功整合公司 SSO 系統後，即**視為符合** `CG-SEC-001`, `CG-SEC-002`, `CG-SEC-005`, `CG-SEC-006` 的規範要求。

- **SSO 整合安全要求**:
  - 應用程式的職責轉變為**安全地實作 SSO 協定**（如 OIDC 或 SAML）。
  - 必須嚴格驗證從 IdP 回傳的 Token 或 Assertion 的簽章與有效性。
  - 必須正確處理登出流程，確保本地 Session 與 IdP Session 均能有效登出。

## 2. 本地帳號密碼實作規範 (Local Account Implementation Guideline)

**注意：本章節規範僅適用於已獲得安全審核批准，確定不使用 SSO 而必須實作本地帳號系統的情境。**

### CG-SEC-001: 【強制】密碼必須經過雜湊加鹽處理 (bcrypt)
- **規範**: 必須使用 **bcrypt** 演算法儲存密碼。嚴禁儲存明文或使用 MD5/SHA-1。

### CG-SEC-002: 【強制】新密碼必須符合複雜度要求
- **規範**:
  1.  最小長度: 8 個字元。
  2.  必須包含至少三種字元：英文大寫、英文小寫、數字、特殊符號。

### CG-SEC-005: 【強制】實作登入失敗鎖定機制
- **規範**:
  1.  當使用者登入失敗時，`failedLoginAttempts` 計數器**必須**增加。
  2.  當計數器達到 **5** 次時，`isLocked` 旗標**必須**被設為 `true`。
  3.  當使用者登入成功時，`failedLoginAttempts` 計數器**必須**被重設為 0。

### CG-SEC-006: 【建議】實作密碼定期變更機制
- **規範**:
  1.  在使用者成功驗證密碼後，**必須**檢查當前時間與 `passwordLastChanged` 時間的差距。
  2.  若差距**大於 90 天**，則拒絕該次登入請求，並強制使用者變更密碼。

### CG-SEC-004: 【強制】所有原則閾值必須可配置
- **規範**: 所有策略的數值**不得**寫死在程式碼中 (Hard-coded)。
- **必須**將下列數值定義在設定檔 (`application.properties` 或 `config.yml`) 中：
  - `security.policy.password.expires-after-days=90`
  - `security.policy.login.max-failed-attempts=5`
