# 後端專案 Rules 模板說明

## 概述

本文件說明如何使用後端專案 Rules 模板系統，為不同的後端專案建立一致的安全規範和開發標準。此模板系統專注於**安全實作規範**和**組態管理原則**，確保後端應用程式在身份認證、資料保護和系統安全方面符合企業級標準。

## 模板結構

```
ai-sdlc/docs_template/backend/rules/
├── design_account_guidline.md          # 身份認證與帳號管理安全規範
└── secure_code_guidline.md             # 應用程式安全與組態管理規範
```

## 第一部分：身份認證與帳號管理規範

### 核心認證策略 (`design_account_guidline.md`)

**用途**: 定義身份認證系統的架構設計和安全實作標準

**主要內容**:

- **SSO 整合優先策略**: 優先採用企業統一單一登入系統
- **本地帳號實作規範**: 當無法使用 SSO 時的密碼管理標準
- **安全原則實作**: 密碼雜湊、複雜度要求、登入鎖定機制
- **組態管理**: 安全參數的可配置化設計
- **合規性要求**: 符合企業安全政策的實作標準

**核心規範重點**:

- **CG-SEC-000**: 強制優先採用 SSO 整合，避免自行管理密碼
- **CG-SEC-001**: 使用 bcrypt 進行密碼雜湊加鹽處理
- **CG-SEC-002**: 密碼複雜度要求（8字元以上，包含三種字元類型）
- **CG-SEC-005**: 登入失敗鎖定機制（5次失敗後鎖定）
- **CG-SEC-006**: 密碼定期變更機制（90天強制變更）
- **CG-SEC-004**: 所有安全參數必須可配置，禁止硬編碼

**適用情境**:

- 企業級後端應用程式開發
- 需要身份認證的 Web 服務
- API 服務的安全實作
- 微服務架構中的認證服務

## 第二部分：應用程式安全與組態管理

### 組態與環境變數管理 (`secure_code_guidline.md`)

**用途**: 規範應用程式的組態管理和環境變數使用標準

**主要內容**:

- **組態安全原則**: 禁止硬編碼敏感資訊
- **環境變數管理**: 外部服務連線的配置方式
- **部署彈性**: 確保應用程式在不同環境間的靈活性
- **風險控制**: 避免因硬編碼導致的安全風險

**核心規範重點**:

- **CG-SEC-008**: 嚴禁在程式碼中硬編碼 IP 位址或主機名稱
- **外部服務連線**: 所有連線資訊必須透過組態檔讀取
- **環境適應性**: 確保應用程式在不同部署環境中的彈性
- **風險管理**: 避免因硬編碼導致的部署錯誤和管理混亂

**技術實作範例**:

```properties
# Spring Boot application.properties 範例
external.service.url=http://api.example.com/v1/data
database.host=localhost
database.port=5432
security.policy.password.expires-after-days=90
security.policy.login.max-failed-attempts=5
```

## 如何使用模板

### 1. 建立新專案的安全規範

1. **複製規範文件**: 將 `ai-sdlc/docs_template/backend/rules/` 複製到新專案的 `.cursor/rules/` 目錄
2. **客製化安全參數**: 根據專案需求調整安全策略的數值
3. **整合 SSO 評估**: 評估是否可以使用企業 SSO 系統
4. **實作組態管理**: 建立適當的組態檔案結構

### 2. 安全規範實作指導

#### SSO 整合優先策略

- **評估階段**: 優先評估企業 SSO 系統的可用性
- **責任轉移**: 採用 SSO 後，將密碼管理責任轉移給 IdP
- **協定實作**: 安全實作 OIDC 或 SAML 協定
- **Token 驗證**: 嚴格驗證 IdP 回傳的 Token 有效性

#### 本地帳號系統實作

- **密碼雜湊**: 使用 bcrypt 演算法進行密碼處理
- **複雜度檢查**: 實作密碼複雜度驗證邏輯
- **鎖定機制**: 實作登入失敗計數和帳號鎖定
- **定期變更**: 實作密碼過期檢查和強制變更

#### 組態管理最佳實踐

- **環境分離**: 為不同環境建立獨立的組態檔案
- **敏感資訊**: 使用環境變數管理敏感資訊
- **版本控制**: 將組態範本納入版本控制，排除實際敏感值
- **部署自動化**: 在部署流程中自動注入環境特定組態

### 3. 規範實作範例

以 Spring Boot 專案為例：

**安全組態範例**:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Value("${security.policy.login.max-failed-attempts:5}")
    private int maxFailedAttempts;
    
    @Value("${security.policy.password.expires-after-days:90}")
    private int passwordExpiresAfterDays;
    
    // 實作安全配置
}
```

**組態檔案範例**:

```properties
# application.properties
security.policy.password.expires-after-days=90
security.policy.login.max-failed-attempts=5
external.service.url=${EXTERNAL_SERVICE_URL:http://localhost:8080}
database.url=${DATABASE_URL:jdbc:postgresql://localhost:5432/mydb}
```

### 4. 維護和更新

- **定期安全審核**: 定期檢查安全規範的實作狀況
- **版本控制**: 將安全規範納入版本控制，確保團隊同步
- **合規性檢查**: 建立自動化檢查機制，確保程式碼符合安全規範
- **團隊培訓**: 確保開發團隊了解並遵循安全規範

## 模板特色

### 企業級安全標準

每個規範都基於企業級安全要求設計：

- **強制性規範**: 標示為【強制】的規範必須嚴格遵循
- **建議性規範**: 標示為【建議】的規範可根據專案需求調整
- **合規性導向**: 確保實作符合企業安全政策要求
- **風險控制**: 明確識別和防範安全風險

### 技術棧適應性

- **多框架支援**: 適用於 Spring Boot、Node.js、Python Django 等
- **資料庫彈性**: 支援各種關聯式和 NoSQL 資料庫
- **認證方案**: 支援多種認證框架和協定
- **部署環境**: 適應不同的雲端和本地部署需求

### 實作指導完整性

- **具體範例**: 提供實際的程式碼範例和組態範本
- **風險說明**: 詳細說明不遵循規範的風險和後果
- **最佳實踐**: 提供業界認可的安全實作最佳實踐
- **檢查清單**: 包含實作完成後的驗證檢查項目

## 最佳實踐

1. **安全優先**: 在專案初期就建立安全規範，避免後續重構
2. **團隊共識**: 確保整個開發團隊了解並遵循安全規範
3. **自動化檢查**: 整合靜態程式碼分析工具，自動檢查安全規範遵循狀況
4. **定期審核**: 定期進行安全審核和規範更新
5. **文件同步**: 保持安全規範文件與實際實作的同步
6. **持續改進**: 根據安全威脅演進和業界最佳實踐持續改進規範

## 注意事項

1. **規範強制性**: 【強制】標記的規範必須嚴格遵循，不得妥協
2. **安全優先**: 安全考量應優先於開發便利性
3. **合規要求**: 確保實作符合相關法規和企業政策要求
4. **風險評估**: 在調整規範前，必須進行充分的安全風險評估
5. **團隊協作**: 安全規範的制定和執行需要整個開發團隊的參與
