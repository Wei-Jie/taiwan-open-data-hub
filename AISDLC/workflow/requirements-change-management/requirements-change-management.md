基於 AISDLC 框架設計的需求變更管理流程，專門處理現有需求的更新、修改和調整，確保文檔間的一致性和追蹤鏈的完整性。

# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**需求變更管理工作流程 (Requirements Change Management)**

## 描述
此工作流程專門處理對現有需求的變更請求，包括新增功能、修改功能、刪除功能和優先級調整。核心特色是基於 PRD → FRD → SRD → AT 追蹤鏈進行影響分析，確保所有變更都能正確更新相關文檔，並維持文檔間的一致性。

## 適用場景
- **使用時機**：收到現有功能的變更請求、需要調整已定義需求時
- **適用專案**：有既存PRD/FRD/SRD/AT文檔的專案，需要進行需求調整
- **執行頻率**：按需執行，當有變更請求時觸發

## 觸發條件
- 利害關係人提出功能變更需求（文字、截圖或混合輸入）
- 現有需求與新發現的業務需求產生衝突
- 技術限制導致需要調整原有需求設計
- Sprint 執行過程中發現需求不完整或有誤

# 角色與責任
# =============================================================================

## 主要負責人 (Primary Owner)
**Agent 角色**：SA Agent (Amanda - System Analyst)
**責任**：負責變更請求分析、影響評估、文檔更新協調和一致性驗證

## 參與者 (Participants)
- **BA Agent (Beatrice)**：協助影響分析、利害關係人溝通、變更記錄管理
- **PM/PO Agent (Victoria)**：評估業務價值、確認優先級、審核PRD層級變更
- **SD Agent (Marcus)**：評估技術可行性、分析SRD層級影響、提供技術建議
- **QA Agent (Quincy)**：分析測試影響、更新AT測試案例、評估回歸測試需求
- **人類用戶**：提供變更請求、確認變更理解、批准最終變更

## 審查者 (Reviewers)
- **人類用戶**：審查變更影響分析、確認文檔更新、批准變更實施
- **PM/PO Agent (Victoria)**：審查業務邏輯一致性和優先級合理性
- **SD Agent (Marcus)**：審查技術設計一致性和架構完整性

# =============================================================================
# 輸入與前置條件
# =============================================================================

## 必要輸入文檔
- **變更請求**：文字描述、截圖或混合輸入的變更需求
- **現有文檔**：相關的PRD、FRD、SRD、AT文檔
- **專案背景資料**：當前Sprint狀態、專案時程、資源限制

## 前置條件
- 專案已有建立的PRD/FRD/SRD/AT文檔體系
- 現有文檔的追蹤鏈關係明確
- 主要利害關係人可配合變更確認和決策
- 有充分時間進行影響分析和文檔更新

## 所需資源
- **工具和系統**：文檔協作平台、版本控制系統、需求管理工具
- **時間資源**：1-3個工作日，依變更複雜度而定
- **人力資源**：相關領域專家、技術團隊代表、業務代表

# =============================================================================
# 執行步驟
# =============================================================================

## 步驟 1：變更請求接收與解析
**描述**：接收各種形式的變更請求並進行初步分類和影響識別

**執行者**：SA Agent (主導), BA Agent (支援)

**具體作業**：
1. **變更請求分類與編號**
   - 文字變更請求：編號為 CHG-TXT-YYYY-MM-DD-XXX
   - 截圖變更請求：編號為 CHG-SCR-YYYY-MM-DD-XXX
   - 混合輸入請求：編號為 CHG-MIX-YYYY-MM-DD-XXX

2. **變更類型識別**
   - 新增功能 (Feature Addition)
   - 修改功能 (Feature Modification)  
   - 刪除功能 (Feature Removal)
   - 優先級調整 (Priority Change)
   - 技術變更 (Technical Change)

3. **初步影響識別**
   - 識別可能受影響的文檔類型（PRD/FRD/SRD/AT）
   - 標記變更的緊急程度和複雜度
   - 準備人機協作確認

**【人機協作確認點1】變更理解與範圍確認**：
- **暫停流程**，呈現變更解析結果
- **呈現內容**：
  - 變更請求編號和分類
  - 變更類型和描述
  - 初步影響範圍推測
  - 變更複雜度評估
- **人類需要確認**：
  1. 變更類型識別是否正確？
  2. 變更描述是否完整準確？
  3. 是否有遺漏的相關變更需求？
  4. 變更的優先級和時程要求？
- **確認選項**：
  - A. 理解正確，繼續分析
  - B. 需要修正理解，請提供澄清
  - C. 需要補充更多變更細節

**檢查點**：
- [ ] 變更請求已正確分類和編號
- [ ] 變更類型已明確識別
- [ ] 人類已確認變更理解正確

**品質標準**：
- 變更描述清晰具體，無歧義
- 變更類型分類準確
- 所有相關變更需求都已收集

---

## 步驟 2：追蹤鏈影響分析
**描述**：基於PRD → FRD → SRD → AT追蹤鏈分析變更的完整影響範圍

**執行者**：SA Agent (主導), BA Agent (支援)

**具體作業**：
1. **追蹤鏈起點識別**
   - 判斷變更從哪個文檔層級開始（PRD/FRD/SRD）
   - 識別變更的根本原因（業務需求/功能調整/技術限制）

2. **向下游影響分析**
   - 如果PRD變更：分析對FRD、SRD、AT的影響
   - 如果FRD變更：分析對SRD、AT的影響  
   - 如果SRD變更：分析對AT的影響

3. **向上游一致性檢查**
   - 確認變更與上游文檔的一致性
   - 識別需要同步更新的上游內容

4. **受影響文檔清單**
   - 列出所有需要更新的文檔
   - 標註具體受影響的章節和內容
   - 評估變更的複雜度和工作量

**【人機協作確認點2】影響範圍確認**：
- **暫停流程**，呈現影響分析結果
- **呈現內容**：
  - 追蹤鏈影響圖示
  - 受影響文檔清單和具體章節
  - 變更複雜度和工作量評估
  - 潛在風險和依賴關係
- **人類需要確認**：
  1. 影響範圍分析是否完整？
  2. 是否有遺漏的影響項目？
  3. 變更的風險和複雜度評估是否合理？
  4. 是否需要調整變更範圍？
- **確認選項**：
  - A. 影響分析正確，繼續評估
  - B. 需要補充影響分析
  - C. 變更範圍過大，需要拆分
  - D. 變更風險過高，考慮取消

**檢查點**：
- [ ] 追蹤鏈影響分析完成
- [ ] 所有受影響文檔已識別
- [ ] 變更複雜度已評估
- [ ] 人類已確認影響範圍

**品質標準**：
- 影響分析涵蓋完整的追蹤鏈
- 所有受影響的文檔和章節都已識別
- 風險評估準確合理

---

## 步驟 3：多Agent協作評估
**描述**：各專業Agent從不同角度評估變更的可行性和影響

**執行者**：多Agent協作 (SA主導協調)

**具體作業**：
1. **PM/PO Agent評估**：
   - 評估變更對Sprint目標的影響（PRD層級）
   - 確認業務價值和投資回報
   - 檢查與產品策略的一致性
   - 評估對利害關係人的影響

2. **SA Agent評估**：
   - 評估功能需求變更的邏輯性（FRD層級）
   - 分析User Stories和AC的調整需求
   - 確認領域知識的一致性
   - 評估對使用者體驗的影響

3. **SD Agent評估**：
   - 評估技術可行性和實作複雜度（SRD層級）
   - 分析對系統架構的影響
   - 評估API和資料模型的調整需求
   - 估算技術實作工作量

4. **QA Agent評估**：
   - 評估對測試策略的影響（AT層級）
   - 識別需要更新的測試案例
   - 評估回歸測試範圍
   - 估算測試工作量

**【人機協作確認點3】可行性與優先級確認**：
- **暫停流程**，呈現多Agent評估結果
- **呈現內容**：
  - 各Agent的評估結果和建議
  - 業務價值vs技術複雜度分析
  - 資源需求和時程估算
  - 風險和依賴關係評估
- **人類需要確認**：
  1. 各專業角度的評估是否合理？
  2. 變更的優先級如何？
  3. 是否在可接受的時程和資源範圍內？
  4. 是否批准進行變更實施？
- **確認選項**：
  - A. 批准變更，繼續執行
  - B. 批准但調整範圍或時程
  - C. 暫緩變更，需要更多資訊
  - D. 拒絕變更，風險或成本過高

**檢查點**：
- [ ] 所有Agent評估完成
- [ ] 業務價值和技術可行性已評估
- [ ] 資源和時程需求已確認
- [ ] 人類已做出變更決策

**品質標準**：
- 評估涵蓋業務、技術、測試各面向
- 工作量估算合理可信
- 風險識別完整準確

---

## 步驟 4：文檔更新執行
**描述**：按照追蹤鏈順序更新相關文檔內容

**執行者**：各Agent協作 (依文檔責任分工)

**具體作業**：
1. **PRD更新** (PM/PO Agent主導)：
   - 更新Sprint目標和範圍（如適用）
   - 調整關鍵交付成果
   - 更新風險和依賴關係
   - 修正業務目標和成功標準

2. **FRD更新** (SA Agent主導)：
   - 新增、修改或刪除User Stories (US-XXX)
   - 更新對應的Acceptance Criteria (AC-XXX-X)
   - 調整功能優先級和依賴關係
   - 更新業務規則和限制條件

3. **SRD更新** (SD Agent主導)：
   - 更新系統架構和設計
   - 修正API規格和資料模型
   - 調整技術實作規範
   - 更新非功能需求

4. **AT更新** (QA Agent主導)：
   - 更新測試案例 (AT-XXX-XX)
   - 修正測試數據和環境需求
   - 調整測試策略和範圍
   - 更新驗收標準的可測試性

**【人機協作確認點4】文檔更新內容確認**：
- **暫停流程**，呈現所有文檔更新內容
- **呈現內容**：
  - 各文檔的具體更新內容對比
  - 新增、修改、刪除的項目清單
  - 編號和引用關係的變更
  - 版本控制和變更記錄
- **人類需要確認**：
  1. 文檔更新內容是否正確反映變更需求？
  2. 編號和引用關係是否正確？
  3. 是否有遺漏或錯誤的更新？
  4. 版本記錄是否完整？
- **確認選項**：
  - A. 文檔更新正確，繼續驗證
  - B. 需要修正部分更新內容
  - C. 需要重新更新特定文檔

**檢查點**：
- [ ] 所有受影響文檔已更新
- [ ] 編號和引用關係正確
- [ ] 版本控制記錄完整
- [ ] 人類已確認更新內容

**品質標準**：
- 更新內容準確反映變更需求
- 保持原有文檔格式和結構
- 編號系統一致性維護

---

## 步驟 5：一致性驗證
**描述**：檢查所有更新後文檔的一致性和追蹤鏈完整性

**執行者**：SA Agent (主導), BA Agent (支援)

**具體作業**：
1. **追蹤鏈完整性驗證**：
   - PRD中的需求都有對應的FRD User Stories
   - FRD中的User Stories都有對應的SRD設計
   - FRD中的AC都有對應的AT測試案例
   - SRD的設計支援所有FRD需求

2. **跨文檔引用檢查**：
   - 確認所有跨文檔引用正確（Markdown內部連結）
   - 檢查編號一致性（US、AC、AT編號對應）
   - 驗證引用的內容仍然存在且正確

3. **內容一致性檢查**：
   - 術語和定義在所有文檔中一致
   - 業務規則和限制條件無矛盾
   - 技術規格和功能需求對應

4. **矛盾識別與解決**：
   - 列出所有發現的不一致處
   - 分析矛盾的根本原因
   - 提供解決方案選項

**【人機協作確認點5】一致性驗證確認**：
- **暫停流程**，呈現一致性檢查結果
- **呈現內容**：
  - 追蹤鏈完整性檢查結果
  - 跨文檔引用驗證報告
  - 發現的不一致處和解決方案
  - 整體文檔品質評估
- **人類需要確認**：
  1. 一致性檢查結果是否準確？
  2. 是否接受提出的解決方案？
  3. 是否有其他需要關注的一致性問題？
  4. 文檔品質是否達到要求？
- **確認選項**：
  - A. 一致性驗證通過，繼續記錄
  - B. 需要修正發現的不一致問題
  - C. 需要額外的一致性檢查

**檢查點**：
- [ ] 追蹤鏈完整性已驗證
- [ ] 跨文檔引用已檢查
- [ ] 內容一致性已確認
- [ ] 所有矛盾已解決

**品質標準**：
- 追蹤鏈無斷裂或遺漏
- 所有引用都正確有效
- 文檔間無邏輯矛盾

---

## 步驟 6：變更記錄與批准
**描述**：記錄變更過程和結果，完成最終批准和通知

**執行者**：BA Agent (主導), SA Agent (支援)

**具體作業**：
1. **變更記錄文檔生成**：
   - 使用Change_Request_Template記錄變更請求
   - 使用Change_Impact_Analysis記錄影響分析
   - 更新Change_History_Log記錄變更歷史

2. **關鍵資訊記錄**：
   - 變更請求的原始內容和編號
   - 變更決策的過程和理由
   - 所有更新文檔的版本資訊
   - 參與決策的人員和時間

3. **變更通知準備**：
   - 準備給利害關係人的變更通知
   - 整理變更對專案的影響摘要
   - 提供更新後文檔的存取連結

4. **最終批准確認**：
   - 確認所有必要批准已獲得
   - 記錄正式批准的時間和人員
   - 完成變更狀態更新

**【人機協作確認點6】最終變更批准**：
- **暫停流程**，呈現完整變更摘要
- **呈現內容**：
  - 變更請求完整摘要
  - 所有更新文檔的清單和版本
  - 變更影響和風險總結
  - 後續執行計劃
- **人類需要確認**：
  1. 變更記錄是否完整準確？
  2. 是否同意正式批准這個變更？
  3. 變更通知內容是否適當？
  4. 是否有其他需要注意的事項？
- **確認選項**：
  - A. 正式批准變更，發送通知
  - B. 批准但需要修改通知內容
  - C. 暫緩批准，需要更多確認

**檢查點**：
- [ ] 變更記錄文檔已生成
- [ ] 關鍵資訊已完整記錄
- [ ] 變更通知已準備
- [ ] 最終批准已獲得

# =============================================================================
# 輸出與交付
# =============================================================================

## 主要交付物
- **更新的專案文檔**：修正後的PRD、FRD、SRD、AT文檔
- **變更請求記錄**：完整的變更請求文檔（Change_Request_Template）
- **變更影響分析報告**：詳細的影響評估和風險分析（Change_Impact_Analysis）
- **變更歷史記錄**：更新的變更歷史日誌（Change_History_Log）
- **一致性驗證報告**：文檔一致性檢查結果和解決方案
- **利害關係人通知**：變更通知和影響說明

## 交付標準
- **完整性**：所有受影響的文檔都已正確更新，無遺漏
- **準確性**：文檔更新內容準確反映變更需求
- **可追蹤性**：維持完整的PRD → FRD → SRD → AT追蹤鏈
- **一致性**：所有文檔間無邏輯矛盾或內容衝突
- **可維護性**：變更記錄完整，便於未來維護和追蹤

## 驗收條件
- [ ] 所有受影響文檔已按追蹤鏈順序更新完成

## Agent 協作規則
- **協作模式**：SA Agent主導協調，各專業Agent提供專業評估，BA Agent負責溝通和記錄
- **溝通協議**：每個確認點都需要相關Agent參與，重要決策需要人類批准
- **衝突解決**：技術爭議由SD Agent決定，業務爭議由PM/PO Agent決定，流程問題由SA Agent協調解決

# =============================================================================

## 品質檢查點
- **檢查點1**：變更理解正確性 - 確保變更需求被正確解析和分類
- **檢查點2**：影響分析完整性 - 確保所有影響都被識別和評估
- **檢查點3**：文檔更新正確性 - 確保文檔更新準確反映變更需求
- **檢查點4**：一致性驗證通過 - 確保文檔間無矛盾且追蹤鏈完整
- **檢查點5**：變更記錄完整性 - 確保變更過程和決策被完整記錄

## 風險控制
- **風險1**：變更範圍擴大失控 - 在每個確認點嚴格控制範圍，必要時拆分變更
- **風險2**：文檔一致性破壞 - 使用系統性驗證流程，確保追蹤鏈完整
- **風險3**：利害關係人溝通不足 - 建立標準通知流程，確保充分溝通
- **風險4**：變更記錄遺失 - 使用標準模板和版本控制，確保記錄完整

## 成功指標
- **指標1**：變更實施成功率達到95%以上，無重大返工
- **指標2**：文檔一致性檢查通過率達到100%
- **指標3**：利害關係人滿意度達到4.0/5.0以上
- **指標4**：變更記錄完整性達到100%，便於後續追蹤
