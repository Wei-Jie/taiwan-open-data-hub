基於 AISDLC 框架設計的需求驗證和正式文檔生成流程，通過分層驗證機制確保需求品質，並生成符合標準的 PRD 和 FRD 文檔。

# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**需求驗證與文檔化工作流程 (Requirements Validation and Documentation)**

## 描述
此工作流程接收來自統一需求提取的初步需求，通過多層次驗證機制確保需求完整性、準確性和可行性，最終生成符合 AISDLC 標準的正式 PRD 和 FRD 文檔。強調人機協作驗證，確保所有關鍵決策都經過人類確認。

## 適用場景
- **輸入來源**：unified-requirements-extraction workflow 的輸出
- **驗證層級**：需求完整性、業務邏輯、技術可行性、優先級排序
- **文檔輸出**：正式的 PRD 和 FRD 文檔，符合 AISDLC 標準

## 觸發條件
- unified-requirements-extraction workflow 完成並交付需求提取報告
- 初步需求文檔需要正式驗證和標準化
- 需要生成可用於後續開發的正式文檔時

- **PM/PO Agent (Victoria)**：提供業務價值判斷、優先級確認、PRD 協作撰寫
- **SD Agent (Marcus)**：提供技術可行性評估、架構建議、SRD 預備工作
- **QA Agent (Quincy)**：提供測試性評估、驗收標準建議
- **人類使用者**：在所有關鍵驗證点提供最終確認和決策

## 審查者 (Reviewers)
- **人類使用者**：審查所有驗證結果和正式文檔
- **PM/PO Agent (Victoria)**：審查 PRD 的業務完整性
- **SD Agent (Marcus)**：審查 FRD 的技術實現可行性

# =============================================================================
# 輸入與前置條件
# =============================================================================

## 必要輸入內容
- **需求提取報告**：來自 unified-requirements-extraction 的完整提取結果
- **人機協作日志**：前期所有確認過程的記錄
- **初步需求文檔**：已建立的需求文檔框架
- **輸入源材料**：原始截图、文字描述等参考材料

## 前置條件
- unified-requirements-extraction workflow 已完成並通過驗收
- 所有關鍵需求都有明确的輸入源追蹤
- 人類使用者可配合進行驗證確認流程
- 相關 Agent 已準備好提供專業意見

## 所需資源
- **文檔模板**：AISDLC 標準的 PRD 和 FRD 模板
- **驗證工具**：需求追蹤矩陣、驗證清單
- **協作平台**：支持多 Agent 協作和人機交互

# =============================================================================
# 執行步骤
# =============================================================================

## 步骤 1：需求完整性驗證
**描述**：系統性檢查需求的完整性，確保所有關鍵方面都已涵蓋

**執行者**：SA Agent (主導)、BA Agent (協助)

**具體作業**：
1. **需求覆蓋度檢查**
   - 功能需求完整性：核心功能、輔助功能、邊界情况
   - 非功能需求檢查：性能、安全、可用性、相容性
   - 使用者角色覆盖：所有使用者類型的需求都已考慮
   - 業務流程完整性：端到端流程無斷點

2. **資訊一致性驗證**
   - 需求間無矛盾衝突
   - 与輸入源資訊一致
   - 术语定义統一

3. **關鍵資訊查缺**
   - 驗收標準定义
   - 邊界條件說明
   - 异常處理機制
   - 依赖关系声明

**【人機協作確認点1】需求完整性確認**：
- **暂停流程**，呈现完整性分析结果
- **呈现内容**：
  - 需求覆蓋度報告：各类需求的完整性状况
  - 缺失專案清單：識別出的遗漏内容
  - 一致性檢查结果：发现的矛盾或不一致
  - 补充建議：推荐增加的需求專案
- **人類需要確認**：
  1. 需求覆盖是否充分？是否有重要遗漏？
  2. 发现的不一致問題如何解决？
  3. 建議补充的需求是否必要？
  4. 当前需求是否足以支撑專案目标？
- **確認選項**：
  1. 確認需求完整，可進入下一步
  2. 需补充[具体需求]，重新驗證
  3. 需解决一致性問題，調整相關需求
  4. 需求范围过大/过小，重新界定

**檢查点**：
- [ ] 需求完整性檢查全面完成
- [ ] **人類使用者已確認需求覆盖充分**
- [ ] 所有不一致問題已解决
- [ ] 补充需求已整合到需求列表

**品质標準**：
- 覆蓋度檢查系統且全面
- **人類確認基于充分的資訊和分析**
- 一致性問題完全解决

---

## 步驟 2：業務邏輯與流程驗證
**描述**：深度驗證業務邏輯的正確性和業務流程的完整性

**執行者**：SA Agent (主導)、BA Agent (業務驗證)、PM/PO Agent (價值確認)

**具體作業**：
1. **業務流程驗證**
   - 端到端流程邏輯檢查
   - 角色权限和职责驗證
   - 業務規則執行檢查
   - 异常場景處理驗證

2. **業務價值確認**
   - 与 PM/PO Agent 協作評估需求价值
   - 確認需求與業務目標的一致性
   - 評估實現成本与价值的匹配度

3. **使用者体验驗證**
   - 使用者操作流程合理性
   - 界面交互邏輯檢查
   - 使用者任务完成的便利性

**【人機協作確認點2】業務邏輯驗證**：
- **暫停流程**，呈現業務驗證結果
- **呈现内容**：
  - 業務流程分析：完整的端到端流程圖和說明
  - 業務規則驗證：關鍵業務邏輯的驗證結果
  - 價值評估報告：需求的業務價值分析
  - 使用者体验分析：使用者操作的便利性評估
- **人類需要確認**：
  1. 業務流程是否符合實際業務操作？
  2. 業務規則理解是否正確？
  3. 需求的業務價值評估是否合理？
  4. 使用者操作流程是否便利可行？
- **確認選項**：
  1. 確認業務邏輯正確，流程合理
  2. 流程邏輯正確但需優化使用者體驗
  3. 業務規則需要調整[具體說明]
  4. 需求价值評估需重新考慮

**檢查点**：
- [ ] 業務流程驗證全面完成
- [ ] **人類使用者已確認業務邏輯正確**
- [ ] 業務價值獲得PM/PO確認
- [ ] 使用者体验問題已解决

**品质標準**：
- 業務邏輯驗證深入且準確
- **流程設計经人類確認可行**
- 價值評估客觀合理

---

## 步骤 3：技术可行性与架构预評估
**描述**：評估需求的技术實現可行性，为后续系統設計做準備

**執行者**：SA Agent (主導)、SD Agent (技术評估)

**具體作業**：
1. **技术實現复杂度評估**
   - 与 SD Agent 協作評估實現难度
   - 識別技术风险和挑战
   - 評估所需技術棧和資源

2. **架构影响分析**
   - 对现有系統架构的影响
   - 新功能的架构設計考慮
   - 性能和扩展性要求分析

3. **實現方案预研**
   - 關鍵技术点的實現思路
   - 第三方依赖和集成需求
   - 開發时程和资源需求估算

**【人機協作確認点3】技术可行性確認**：
- **暂停流程**，呈现技术評估结果
- **呈现内容**：
  - 技术可行性報告：實現难度和风险評估
  - 架构影响分析：对系統架构的影响評估
  - 實現方案建議：關鍵功能的技术實現思路
  - 资源需求評估：開發时间和人力资源估算
- **人類需要確認**：
  1. 技术實現方案是否可行？
  2. 是否存在不可接受的技术风险？
  3. 资源需求評估是否合理？
  4. 是否需要調整需求以降低實現复杂度？
- **確認選項**：
  1. 確認技术可行，可按计划實現
  2. 技术可行但需要额外资源[具体說明]
  3. 存在技术风险，需要調整需求
  4. 部分需求實現困难，建議分阶段交付

**檢查点**：
- [ ] 技术可行性評估全面完成
- [ ] **人類使用者已確認技术方案可行**
- [ ] 技术风险已識別并有应对措施
- [ ] 资源需求估算获得认同

**品质標準**：
- 技术評估專業且客观
- **可行性结论经人類確認认可**
- 风险識別全面且有应对措施

---

## 步驟 4：優先級排序與發佈計畫
**描述**：確定需求優先級，制定合理的實現和發佈計畫

**執行者**：SA Agent (主導)、PM/PO Agent (業務決策)、BA Agent (協調)

**具體作業**：
1. **優先級評估標準**
   - 業務價值影響度
   - 使用者需求紧迫性
   - 技术實現复杂度
   - 依赖关系影响

2. **多維度優先級分析**
   - MoSCoW 分析（Must/Should/Could/Won't）
   - 价值/成本矩陣分析
   - 风险/收益評估
   - 依赖关系考量

3. **發佈計畫建議**
   - MVP (最小可行产品) 范围界定
   - 分階段發佈策略
   - 里程碁和檢查點設定

**【人機協作確認點4】優先級與計畫確認**：
- **暫停流程**，呈現優先級分析結果
- **呈现内容**：
  - 优先级排序结果：P0/P1/P2 分级及詳細理由
  - 多维度分析報告：各种評估方法的结果对比
  - 發布计划建議：分阶段實現的建議方案
  - 依赖关系图：需求间的依赖和影响关系
- **人類需要確認**：
  1. 优先级排序是否符合業務实际需求？
  2. 發布计划是否合理可行？
  3. MVP 范围是否能满足核心業務需求？
  4. 是否需要調整某些需求的优先级？
- **確認選項**：
  1. 確認优先级和计划合理
  2. 优先级合理但计划需調整
  3. 需要重新評估部分需求优先级
  4. MVP 范围需要調整[具体說明]

**檢查点**：
- [ ] 优先级排序全面且合理
- [ ] **人類使用者已確認优先级符合業務需求**
- [ ] 發佈計畫可行且有明確里程碑
- [ ] 依赖关系已充分考慮

**品质標準**：
- 优先级評估多维度且客观
- **排序结果经人類確認符合实际**
- 發布计划切实可行

## 步驟 5：正式文檔生成與標準化
**描述**：基於驗證結果生成符合 AISDLC 標準的正式 PRD 和 FRD 文檔

**執行者**：SA Agent (主導)、PM/PO Agent (PRD協作)

**具體作業**：
1. **PRD 文檔撰寫**
   - 使用 `docs_template/prd/PRD_Sprint_Template.md`
   - 与 PM/PO Agent 協作確保業務完整性
   - 整合所有驗證结果和確認内容

2. **FRD 文檔撰寫**
   - 使用 `docs_template/frd/FRD_Module_Template.md`
   - 詳細描述功能規格和驗收標準
   - 確保技术實現指导清晰

3. **需求追蹤矩陣建立**
   - 建立从輸入源到最終需求的完整追蹤链
   - 記錄所有驗證過程和人類確認
   - 確保可追溯性和变更管理

**【人機協作確認點5】最終文檔確認**：
- **暫停流程**，呈現完整的正式文檔
- **呈现内容**：
  - 完整的 PRD 文檔：符合 AISDLC 標準格式
  - 完整的 FRD 文檔：包含詳細功能規格
  - 需求追蹤矩陣：完整的可追溯链条
  - 驗證記錄摘要：所有驗證過程的汇总
- **人類需要確認**：
  1. PRD 是否完整准确反映業務需求？
  2. FRD 是否提供充分的功能實現指導？
  3. 文檔格式是否符合專案標準？
  4. 是否可以作为正式的開發依据？
- **確認選項**：
  1. 確認文档完整准确，批准發布
  2. 文檔基本正確，需微調[具體內容]
  3. 发现重要遗漏，需要补充完善
  4. 需要重新審查部分內容

**檢查点**：
- [ ] PRD 和 FRD 文檔結構完整規範
- [ ] **人類使用者已確認文檔內容準確**
- [ ] 需求追蹤矩陣完整可靠
- [ ] 所有驗證記錄已整理归档

**品质標準**：
- 文檔格式嚴格符合 AISDLC 標準
- **內容準確性經人類最終確認**
- 追蹤链条完整无断点

# 協作与整合
# =============================================================================

## 前置 Workflows
- **unified-requirements-extraction**：提供需求提取報告和初步需求文檔

## 后续 Workflows  
- **user-story-and-design**：基于正式 PRD/FRD 進行使用者故事映射和系統設計

## 并行 Workflows
*无并行 workflow，需要完成全部驗證后才能進入后续流程*

## Agent 協作规则
- **協作模式**：SA Agent 主導驗證，各專業 Agent 提供领域专长，人類使用者提供最終確認
- **文檔協作**：SA 與 PM/PO 共同撰寫 PRD，SA 獨立完成 FRD
- **驗證協作**：多 Agent 从不同角度進行驗證，確保全面性
- **人機協作協議**：關鍵驗證点必须暂停等待人類確認，記錄所有決策過程

# =============================================================================
# 品质控制与监控
# =============================================================================

## 品质檢查点
- **每个驗證步骤完成后**：檢查驗證的全面性和准确性
- **文檔生成完成後**：檢查格式規範性和內容完整性
- **最終交付前**：進行完整的品質審查

## 风险控制
- **驗證不充分**：建立檢查清單，確保所有方面都得到驗證
- **文檔品質問題**：使用標準模板，進行格式和內容雙重檢查
- **人機協作不足**：確保關鍵点都有人類参与，記錄完整
- **Agent 協作衝突**：建立明确的责任分工和決策機制

## 成功指标
