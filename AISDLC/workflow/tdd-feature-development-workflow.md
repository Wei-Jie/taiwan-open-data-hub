# TDD 功能開發工作流程

## Workflow 名稱
**TDD 功能開發工作流程 (Test-Driven Development Workflow)**

## 描述
此工作流程實現完整的測試驅動開發循環，確保**單一功能**開發嚴格遵循 TDD 紅燈→綠燈→重構循環，並完全符合需求規格文檔要求。

**⚠️ 重要限制：此 workflow 目前專注於單一功能的開發，不適用於多功能並行開發或大型模組開發。**

## 適用場景
- **使用時機**：當單一 User Story 和其 Acceptance Criteria 已明確定義時
- **適用專案**：單一新功能開發、單一功能增強、單一缺陷修復
- **執行頻率**：每個單一開發任務執行
- **不適用**：多功能並行開發、整個模組重構、大型系統開發

## 觸發條件
- User Story 已完成並包含明確的 Acceptance Criteria
- 相關的 PRD/FRD/SRD 文檔已存在且經過審核
- 開發環境已設置並可用
- 相關的技術規格和架構指導已明確

# =============================================================================
# 角色與責任
# =============================================================================

## 主要負責人 (Primary Owner)
**Agent 角色**：Dev Agent  
**責任**：執行完整的 TDD 開發循環、確保代碼品質、協調整個開發流程

## 參與者 (Participants)
- **QA Agent**：檢查測試案例完整性、生成測試報告
- **SA Agent**：需求澄清、確認功能實現符合原始需求
- **SD Agent**：架構指導、代碼品質審查、確保符合技術規範

## 審查者 (Reviewers)
- **QA Agent**：測試執行結果、品質報告
- **SD Agent**：代碼架構合規性、技術實現品質

# =============================================================================
# 輸入與前置條件
# =============================================================================

## 必要輸入文檔
- **User Story**：包含明確的 Acceptance Criteria
- **PRD/FRD/SRD**：產品需求、功能需求、系統技術需求文檔
- **Developer Guideline**：專案編碼規範和最佳實踐

## 前置條件
- 開發環境完整設置（IDE、測試框架、Linter 等）
- 專案依賴套件已安裝且版本正確
- 測試環境和相關服務已可用

## 所需資源
- **工具和系統**：測試框架、Linter 工具、代碼覆蓋率工具、CI/CD 流水線
- **時間資源**：根據功能複雜度，通常為 1-5 個工作日
- **人力資源**：跨功能團隊協作 (Dev, QA, SA, SD)

# =============================================================================
# 執行步驟
# =============================================================================

## 步驟 1：需求追蹤與文檔驗證
**描述**：執行需求追蹤矩陣分析，確保用戶需求精確對應到專案功能

**執行者**：SA Agent (主導) + Dev Agent (協作)

**具體作業**：
1. 執行 requirement-traceability-analysis workflow
2. 建立完整的 PRD→FRD→SRD→AT 需求追蹤鏈
3. 檢查必要文檔存在性 (User Story、PRD/FRD/SRD、Developer Guideline)
4. 確認測試文檔和模板可用性

**檢查點**：
- [ ] 需求追蹤鏈已建立且完整
- [ ] 用戶已確認功能對應的正確性
- [ ] 所有必要文檔已確認存在
- [ ] 開發規範和品質標準已明確

**品質標準**：
- 用戶需求解析準確率 100%
- 需求追蹤鏈完整且無遺漏
- 所有開發標準都有明確依據

---

## 步驟 2：規格驗證與需求理解確認
**描述**：建立規格驗證閘門，確保需求理解完全準確，防止錯誤推測

**執行者**：Dev Agent (主導) + SA Agent (協作) + QA Agent (驗證)

**具體作業**：
1. 識別並消除規格歧義點，禁止基於假設進行推測
2. 建立規格點分解與編號 (SPEC-001, SPEC-002...)
3. 分析現有測試覆蓋與規格對齊情況
4. 進行多 Agent 交叉驗證 (SA+QA+Dev) 確保理解一致

**檢查點**：
- [ ] 所有歧義點已標記並獲得澄清
- [ ] 規格點到實現的映射矩陣已建立
- [ ] 測試與規格的對應關係已確認
- [ ] 三 Agent 交叉驗證已完成且達成一致

**品質標準**：
- 歧義消除率 100%
- 規格映射完整性：每個 AC 都有對應規格點
- 多 Agent 一致性要求達成

---

## 步驟 3：AT 測試文檔準備
**描述**：根據規格驗證結果，準備或創建必要的 AT 測試文檔

**執行者**：QA Agent (主導) + Dev Agent (協作)

**具體作業**：
1. 評估現有 AT 文檔覆蓋情況 (完整/部分/無覆蓋)
2. 按照 AT_Module_Template.md 格式創建或更新測試案例
3. 為每個 Acceptance Criteria 創建對應測試案例
4. 使用 Given-When-Then 格式編寫測試步驟
5. 建立測試案例矩陣確保 100% 覆蓋率

**檢查點**：
- [ ] AT 測試文檔已準備完成
- [ ] 所有 AC 都有對應的測試案例
- [ ] 測試案例矩陣覆蓋率達到 100%
- [ ] 測試環境和測試資料需求已明確

**品質標準**：
- AC 到測試案例的對應率 100%
- 測試案例能追溯到對應的 User Story
- 測試案例描述清晰、可執行且可重複

---

## 步驟 4：TDD 紅燈階段 - 編寫失敗測試
**描述**：基於規格點編寫失敗的測試案例，確保每個測試精確對應規格要求

**執行者**：Dev Agent (主導) + QA Agent (驗證)

**具體作業**：
1. 編寫失敗的單元測試，每個測試必須標記對應的規格點編號
2. 編寫失敗的整合測試，涵蓋規格中定義的所有接口
3. 驗證 AT 測試案例與規格點的一致性
4. 建立測試基礎設施和檔案管理策略

**檢查點**：
- [ ] 所有單元測試和整合測試已編寫且處於失敗狀態
- [ ] 測試執行環境已準備就緒
- [ ] 所有測試的失敗原因符合預期（功能尚未實現）
- [ ] 測試檔案管理策略已建立

**品質標準**：
- 測試案例必須針對具體的功能需求
- 測試覆蓋必須與 AC 完全對應
- 每個測試檔案都必須有完整的元數據註釋

---

## 步驟 5：TDD 綠燈階段 - 實現功能代碼
**描述**：實現功能代碼使測試通過，同時確保實現與規格完全符合

**執行者**：Dev Agent (主導) + SA Agent (規格驗證)

**具體作業**：
1. 編寫最簡單的實現使測試通過，確保實現精確符合規格點要求
2. 進行實時規格符合度驗證，確保實現與規格描述完全一致
3. 執行單元測試和整合測試，確認功能在各層級正常運作
4. 按照 AT 文檔執行驗收測試，驗證所有規格點通過

**檢查點**：
- [ ] 所有單元測試通過（綠燈狀態）
- [ ] 所有整合測試通過
- [ ] 所有 AT 測試案例執行通過
- [ ] 所有 Acceptance Criteria 都被滿足

**品質標準**：
- 測試通過率必須達到 100%
- 功能實現必須完全符合 AC 描述
- 不允許跳過任何失敗的測試

---

## 步驟 6：TDD 重構階段 - 代碼優化
**描述**：優化代碼結構和品質，同時保持所有測試通過

**執行者**：Dev Agent (主導) + SD Agent (協作)

**具體作業**：
1. 改善代碼的可讀性和可維護性，消除重複代碼
2. 應用適當的設計模式，確保代碼符合 SOLID 原則
3. 優化效能瓶頸和演算法複雜度
4. 每次重構後立即執行完整測試套件，確保所有測試通過

**檢查點**：
- [ ] 代碼結構已優化且更易維護
- [ ] 所有測試在重構後仍然通過
- [ ] 代碼風格符合專案規範
- [ ] 沒有引入新的技術債務

**品質標準**：
- 重構過程中測試通過率必須保持 100%
- 代碼覆蓋率不應下降
- 重構後的代碼更符合設計原則和最佳實踐

---

## 步驟 7：品質控制與合規檢查
**描述**：執行全面的品質檢查確保代碼符合所有標準

**執行者**：Dev Agent (主導) + SD Agent (協作)

**具體作業**：
1. 執行 Linter 工具檢查，修正所有錯誤和警告
2. 對照 Developer Guideline 進行合規性檢查
3. 驗證測試覆蓋率達到專案要求（≥ 80%）
4. 檢查安全性和最佳實踐，確認架構合規性

**檢查點**：
- [ ] 所有 linter 檢查通過，無錯誤和警告
- [ ] 代碼完全符合 Developer Guideline
- [ ] 測試覆蓋率達到專案要求
- [ ] 安全性和架構合規性驗證通過

**品質標準**：
- Linter 錯誤數必須為 0
- Coding guideline 符合度必須為 100%
- 測試覆蓋率必須達到專案設定標準

---

## 步驟 8：測試報告生成與環境清理
**描述**：執行 AT 驗收測試、生成測試報告並清理測試環境

**執行者**：QA Agent (主導) + Dev Agent (協作)

**具體作業**：
1. 按照 AT 文檔執行所有驗收測試案例，記錄測試結果
2. 使用 Test_Report_Template.md 格式生成完整測試報告
3. 分析失敗測試案例並進行修正，直到所有測試通過
4. 清理測試環境，移除臨時檔案，保留重要測試資產

**檢查點**：
- [ ] 所有 AT 測試案例已執行並記錄結果
- [ ] 測試報告已生成且使用標準格式
- [ ] 所有失敗測試已修正，測試通過率 100%
- [ ] 測試環境已清理且回到穩定狀態

**品質標準**：
- P0 測試案例通過率必須為 100%
- 測試報告必須使用 Test_Report_Template.md 格式
- 清理操作不得誤刪重要測試檔案

# =============================================================================
# 輸出與交付
# =============================================================================

## 主要交付物
- **通過所有測試的功能代碼**：完全符合 PRD/FRD/SRD 需求的實現代碼
- **AT 測試文檔**：使用 AT_Module_Template.md 格式的測試案例文檔
- **測試報告**：使用 Test_Report_Template.md 格式的完整測試報告
- **代碼覆蓋率報告**：單元測試和整合測試的覆蓋率分析

## 交付標準
- **功能完整性**：所有 User Story 的 Acceptance Criteria 100% 滿足
- **測試通過率**：P0 功能測試通過率 100%，P1 功能測試通過率 ≥ 95%
- **代碼品質**：所有 linter 檢查通過，完全符合 coding guideline
- **測試覆蓋率**：單元測試覆蓋率 ≥ 80%，關鍵業務邏輯覆蓋率 100%

## 驗收條件
- [ ] 所有 Acceptance Criteria 已通過對應的 AT 測試案例驗證
- [ ] 功能在端到端測試中表現正確
- [ ] 回歸測試通過，確認沒有破壞現有功能
- [ ] 所有品質檢查點都已通過
- [ ] 測試報告完整且風險評估合理

## 後續流程交接
- **交接給**：部署和發布流程 (如 CI/CD 流水線)
- **交接內容**：經過完整測試的功能代碼、完整的測試文檔和報告
- **交接標準**：所有測試通過且有詳細記錄、代碼品質符合發布標準

# =============================================================================
# 協作與整合
# =============================================================================

## 前置 Workflows
- **requirement-traceability-analysis**：將用戶需求對應到專案功能並建立完整追蹤鏈（強制前置）
- **unified-requirements-extraction**：提供清晰的 User Story 和初始需求分析
- **user-story-and-design**：提供詳細的 SRD 和技術設計規格

## 後續 Workflows  
- **document-consistency-check**：檢查開發過程中生成的文檔與現有文檔的一致性
- **api-specification-generation**：如功能涉及 API，需要生成或更新 API 規格文檔

## 並行 Workflows
- **代碼品質審查**：基於 Developer_Guideline.md 標準進行持續的代碼品質檢查

## Agent 協作規則
- **協作模式**：Dev Agent 主導，QA Agent 支援測試，SA Agent 提供需求澄清，SD Agent 進行技術審查
- **溝通協議**：關鍵決策點進行協作確認，問題升級機制確保及時解決
- **衝突解決**：需求解釋衝突由 SA Agent 分析，技術實現衝突由 SD Agent 評估，測試策略衝突由 QA Agent 主導

# =============================================================================
# 品質控制與監控
# =============================================================================

## 品質檢查點
- **檢查點 1**：需求追蹤與文檔驗證 - 確保需求追蹤鏈完整且獲得用戶確認
- **檢查點 2**：規格驗證 - 確保所有 Agent 對需求理解一致且準確
- **檢查點 3**：TDD 循環品質 - 確保完整執行紅燈-綠燈-重構循環
- **檢查點 4**：品質合規 - 確保代碼符合所有品質標準和專案規範
- **檢查點 5**：最終交付 - 確保所有交付物完整且符合標準

## 風險控制
- **風險 1：需求理解偏差** - 強制進行需求追蹤分析和多 Agent 交叉驗證
- **風險 2：測試覆蓋不足** - 建立 AC 到測試案例的追蹤矩陣，確保 100% 覆蓋
- **風險 3：TDD 流程不當執行** - 建立強制檢查點，記錄每個階段的執行證據
- **風險 4：品質標準妥協** - 建立不可妥協的品質閘門，問題必須修正才能繼續

## 成功指標
- **指標 1：需求實現準確度** - 95% 的需求在後續階段無需重大修改
- **指標 2：代碼品質指標** - Linter 通過率 100%，測試覆蓋率 ≥ 80%
- **指標 3：TDD 流程執行品質** - 完整的紅燈-綠燈-重構循環執行記錄
- **指標 4：測試效率指標** - P0 測試案例通過率 100%，整體通過率 ≥ 95%
- **指標 5：交付品質指標** - 交付物符合標準格式，風險評估準確合理
