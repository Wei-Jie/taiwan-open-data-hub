# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**前後端交互流程分析 (Frontend-Backend Interaction Analysis)**

## 描述
此工作流程分析現有的 AISDLC 文檔（FRD、SRD、User Stories、API 規格），識別並補充前後端交互流程的描述缺失。主要目標是增強 SRD 文檔，加入完整的業務流程序列圖、API 調用時序、狀態同步機制等內容，為前後端開發團隊提供明確的協作指引。

## 適用場景
- **使用時機**：SRD 文檔已完成基本技術設計，但缺乏詳細的前後端交互描述時
- **適用專案**：包含前端介面和後端 API 的 Web 應用程式、行動應用程式、單頁應用程式
- **執行頻率**：在系統設計階段完成後、開發實作開始前執行

## 觸發條件
- 已完成基本的 SRD 文檔，包含 API 設計和資料模型
- User Stories 和 API 規格文檔已就緒
- 開發團隊反映需要更清晰的前後端協作指引
- 專案複雜度較高，涉及多個業務流程和複雜的狀態管理

# =============================================================================
# 角色與責任
# =============================================================================

## 主要負責人 (Primary Owner)
**Agent 角色**：SD Agent (Marcus - System Designer)
**責任**：負責分析技術交互流程，設計序列圖，更新 SRD 文檔內容

## 參與者 (Participants)
- **SA Agent (Amanda)**：提供業務流程分析，協助識別關鍵交互點
- **Frontend Developer Agent**：提供前端狀態管理和使用者體驗的技術建議
- **Backend Developer Agent**：提供 API 設計和資料流的技術細節
- **PM/PO Agent (Victoria)**：確認業務流程的優先級和核心價值鏈
- **人類用戶**：在關鍵設計決策點提供確認和業務理解指導

## 審查者 (Reviewers)
- **人類用戶**：審查交互流程設計的合理性和完整性
- **Frontend/Backend Developer Agents**：審查技術實現的可行性
- **QA Agent (Quincy)**：從測試角度評估交互流程的可測試性

# =============================================================================
# 輸入與前置條件
# =============================================================================

## 必要輸入文檔
- **現有 SRD 文檔**：需要增強交互描述的系統需求文檔
- **FRD 文檔**：包含詳細 User Stories 和 Acceptance Criteria
- **API 規格文檔**：完整的 API 端點規格和資料結構定義
- **相關 User Stories**：涉及前後端交互的用戶故事

## 前置條件
- SRD 基本架構設計已完成（API 設計、資料模型、系統整合）
- API 規格文檔已生成並與 SRD 保持一致
- User Stories 具備明確的 Acceptance Criteria
- 前後端技術棧已確定

## 所需資源
- **工具和系統**：序列圖繪製工具（Mermaid）、文檔編輯器
- **時間資源**：2-4 小時，視專案複雜度而定
- **人力資源**：需要熟悉前後端技術的系統設計師或架構師

# =============================================================================
# 執行步驟
# =============================================================================

## 步驟 1：現有文檔掃描與分析
**描述**：掃描並分析現有 SRD 文檔，識別缺失的前後端交互描述

**執行者**：SD Agent (主導)

**具體作業**：
1. **SRD 文檔完整性檢查**
   - 檢查現有 SRD 是否包含前後端交互流程章節
   - 識別已描述和未描述的業務流程
   - 評估現有交互描述的完整度和清晰度

2. **關聯文檔分析**
   - 分析 FRD 中的 User Stories 和業務流程
   - 檢查 API 規格與 SRD 的一致性
   - 識別複雜的跨系統交互場景

3. **優先級評估**
   - 識別核心業務流程（如用戶認證、主要功能流程）
   - 評估交互複雜度和對開發的影響
   - 制定分析和更新的優先順序

**檢查點**：
- [ ] 所有相關 SRD 文檔已掃描完畢
- [ ] 缺失的交互描述已識別並分類
- [ ] 優先級和工作範圍已明確定義

**品質標準**：
- 完整性：涵蓋所有涉及前後端交互的模組
- 準確性：正確識別現有描述的不足之處
- 可行性：更新計劃在可執行的範圍內

---

## 步驟 2：核心業務流程提取與建模
**描述**：從 FRD 和 User Stories 中提取核心業務流程，建立交互流程模型

**執行者**：SD Agent (主導)、SA Agent (協作)

**具體作業**：
1. **業務流程識別**
   - 從 User Stories 中提取完整的業務操作流程
   - 識別用戶觸發的操作和系統自動化流程
   - 分析異常情況和錯誤處理流程

2. **交互點映射**
   - 識別每個業務流程中的前後端交互點
   - 確定 API 調用的觸發時機和條件
   - 分析資料流向和狀態變化

3. **流程層級劃分**
   - **主要流程**：用戶核心操作的完整流程
   - **子流程**：可復用的業務邏輯片段
   - **異常流程**：錯誤處理和異常情況

4. **相依性分析**
   - 識別流程間的依賴關係
   - 分析並行執行和順序執行的需求
   - 確定關鍵路徑和性能瓶頸點

**【人機協作確認點1】核心流程優先級確認**：
- **暫停流程**，呈現業務流程分析結果
- **呈現內容**：
  - 識別的核心業務流程清單
  - 流程複雜度和優先級評估
  - 建議的分析順序和重點

**確認要點**：
- 核心業務流程是否識別完整？
- 優先級排序是否符合業務重要性？
- 是否有特別需要關注的交互場景？

**檢查點**：
- [ ] 主要業務流程已完整提取
- [ ] 交互點和資料流向已清晰識別
- [ ] 流程依賴關係已分析完成

**品質標準**：
- 完整性：涵蓋所有重要的用戶操作流程
- 準確性：交互點識別與實際業務邏輯一致
- 清晰性：流程描述便於技術團隊理解

---

## 步驟 3：交互序列圖設計與 API 時序規劃
**描述**：為核心業務流程設計詳細的交互序列圖和 API 調用時序

**執行者**：SD Agent (主導)、Frontend/Backend Developer Agents (協作)

**具體作業**：
1. **序列圖結構設計**
   - 定義參與者：用戶、前端應用、後端 API、資料庫、外部服務
   - 設計交互步驟的邏輯順序
   - 標注關鍵的決策點和分支邏輯

2. **Mermaid 序列圖實作**
   - 使用標準的 Mermaid 語法繪製序列圖
   - 包含正常流程和主要異常情況
   - 標注 API 端點和重要的資料結構

3. **API 調用時序設計**
   - **前置檢查階段**：前端驗證、權限檢查、狀態驗證
   - **核心調用階段**：主要 API 請求、依賴 API、並行請求
   - **後續處理階段**：狀態更新、錯誤處理、用戶回饋

4. **依賴關係文檔化**
   - 明確 API 間的調用依賴
   - 定義並行調用的可行性
   - 規劃錯誤情況下的回滾機制

5. **文檔一致性驗證**
   - **FRD 一致性檢查**：驗證序列圖與 FRD 中 User Stories 的邏輯一致性
   - **API 規格對比**：確認 API 調用時序與現有 API 規格文檔的匹配性
   - **AT 覆蓋性檢查**：檢查異常處理流程是否有對應的 AT 測試案例
   - **PRD 影響分析**：識別交互設計可能影響的 PRD 功能描述
   - **不一致問題記錄**：詳細記錄發現的文檔間矛盾和缺失

**檢查點**：
- [ ] 核心流程的序列圖已完成繪製
- [ ] API 調用時序已詳細規劃
- [ ] 異常情況的處理流程已考慮
- [ ] 與 FRD、API規格、AT 的一致性已驗證
- [ ] 文檔不一致問題已識別並記錄

**品質標準**：
- 完整性：序列圖涵蓋完整的業務流程
- 技術準確性：API 調用符合既有的 API 規格
- 可讀性：序列圖清晰易懂，便於開發團隊參考
- 一致性：與所有相關文檔保持邏輯一致

---

## 步驟 4：前端狀態管理與同步策略設計
**描述**：設計前端狀態管理策略和與後端的資料同步機制

**執行者**：Frontend Developer Agent (主導)、SD Agent (協作)

**具體作業**：
1. **狀態結構設計**
   - 定義模組化的前端狀態結構
   - 設計資料、載入狀態、錯誤狀態的管理
   - 規劃狀態的持久化策略（localStorage、sessionStorage）

2. **同步策略規劃**
   - **初始載入**：頁面載入時的資料獲取策略
   - **增量更新**：資料變更時的部分更新邏輯
   - **即時同步**：需要實時更新的資料項目（WebSocket、Server-Sent Events）
   - **離線處理**：網路中斷時的本地資料處理

3. **快取策略設計**
   - 前端資料快取的範圍和生命週期
   - 快取失效的觸發條件
   - 快取與後端資料的一致性保證

4. **效能優化策略**
   - 批次請求和請求合併
   - 分頁載入和虛擬滾動
   - 預載入和懶載入策略

**檢查點**：
- [ ] 前端狀態結構已合理設計
- [ ] 資料同步策略已完整規劃
- [ ] 快取和效能優化策略已考慮

**品質標準**：
- 可維護性：狀態管理結構清晰，便於維護
- 效能合理性：同步策略平衡效能和資料一致性
- 用戶體驗：載入和錯誤處理提供良好的用戶體驗

---

## 步驟 5：異常處理與容錯機制設計
**描述**：設計完整的異常處理和容錯機制，確保系統穩定性

**執行者**：SD Agent (主導)、QA Agent (協作)

**具體作業**：
1. **異常情況分類**
   - **網路異常**：連線超時、網路中斷、請求失敗
   - **業務異常**：權限不足、資料衝突、驗證失敗
   - **系統異常**：伺服器錯誤、服務不可用、資料庫異常

2. **容錯機制設計**
   - **重試邏輯**：指數退避重試策略
   - **降級方案**：部分功能不可用時的替代方案
   - **斷路器模式**：防止系統雪崩的保護機制

3. **用戶體驗保障**
   - **載入狀態顯示**：明確的載入指示器
   - **錯誤訊息設計**：使用者友善的錯誤提示
   - **操作引導**：錯誤發生時的操作建議

4. **監控與日誌**
   - 前端錯誤監控和上報機制
   - 關鍵交互點的日誌記錄
   - 異常情況的追蹤和分析

**【人機協作確認點2】關鍵交互設計確認**：
- **暫停流程**，呈現交互設計結果
- **呈現內容**：
  - 核心業務流程的序列圖設計
  - 前端狀態管理和同步策略
  - 異常處理和容錯機制設計

**確認要點**：
- 序列圖是否準確反映業務流程？
- 狀態管理策略是否合理可行？
- 異常處理是否涵蓋主要風險點？

**檢查點**：
- [ ] 異常情況已全面識別和分類
- [ ] 容錯機制已合理設計
- [ ] 用戶體驗保障措施已完整規劃

**品質標準**：
- 全面性：涵蓋主要的異常情況和風險點
- 實用性：容錯機制可實際執行和測試
- 用戶導向：異常處理優先考慮用戶體驗

---

## 步驟 6：文檔更新與一致性修復
**描述**：將分析結果整合到 SRD 文檔中，並自動修復相關文檔的一致性問題

**執行者**：SD Agent (主導)、SA Agent (協作)

**具體作業**：

### 6.1 SRD 文檔更新
1. **SRD 章節更新**
   - 在 SRD 的「前後端交互流程」章節中加入設計內容
   - 更新業務流程序列圖
   - 補充 API 調用時序和依賴說明
   - 加入前端狀態管理和異常處理描述

2. **內容結構化**
   - 使用統一的章節結構和格式
   - 確保 Mermaid 序列圖語法正確
   - 標注與其他文檔的引用關聯

### 6.2 相關文檔一致性修復
1. **PRD 文檔修復**
   - **功能描述修正**：根據交互分析結果更新功能描述中的前後端協作說明
   - **業務流程更新**：修正 PRD 中業務流程描述與實際交互序列的不一致
   - **用戶體驗說明補充**：加入交互設計對用戶體驗的影響說明

2. **FRD 文檔修復**
   - **User Stories 更新**：修正與前後端交互不符的 User Stories 描述
   - **業務規則修正**：更新業務規則以反映正確的交互時序
   - **驗收條件增強**：加入前後端交互相關的驗收條件

3. **API 規格文檔修復**
   - **調用時序修正**：更新 API 規格中的調用順序說明
   - **依賴關係更新**：修正 API 間的依賴關係描述
   - **錯誤處理增強**：補充 API 錯誤處理與前端容錯的協調機制

4. **AT 測試案例修復**
   - **測試場景補充**：加入前後端交互異常的測試案例
   - **測試步驟修正**：更新測試步驟以反映正確的交互流程
   - **驗收標準同步**：確保 AT 驗收標準與修正後的文檔一致

### 6.3 修復確認與驗證
1. **修復內容摘要**
   - 生成詳細的文檔修復報告
   - 列出所有修正的不一致問題
   - 標注修復的影響範圍和重要程度

2. **交叉驗證檢查**
   - 驗證修復後各文檔間的一致性
   - 檢查修復是否引入新的不一致問題
   - 確認修復符合 AISDLC 文檔標準

**【人機協作確認點3】SRD 更新內容確認**：
- **暫停流程**，呈現 SRD 更新內容
- **呈現內容**：
  - 更新後的 SRD 文檔相關章節
  - 新增的序列圖和技術描述
  - 與其他文檔的一致性檢查結果

**確認要點**：
- SRD 更新內容是否完整準確？
- 技術描述是否便於開發團隊理解和執行？
- 是否需要進一步調整或補充？

**【人機協作確認點4】文檔修復確認**：
- **暫停流程**，呈現文檔修復計劃和結果
- **呈現內容**：
  - 檢測到的各文檔不一致問題清單
  - 擬定的修復方案和影響範圍
  - 修復後的文檔變更摘要

**確認要點**：
- 檢測到的不一致問題是否需要修復？
- 修復方案是否適當且不會引入新問題？
- 是否同意執行文檔修復更新？

**檢查點**：
- [ ] 所有 SRD 文檔已完成更新
- [ ] 內容格式和結構已標準化
- [ ] 相關文檔的一致性問題已識別和修復
- [ ] PRD/FRD/AT/API 規格文檔已同步更新
- [ ] 文檔修復報告已生成並確認

**品質標準**：
- 完整性：SRD 文檔包含完整的前後端交互描述
- 準確性：技術內容與實際系統設計保持一致
- 實用性：描述內容能有效指導開發實作
- 一致性：所有相關文檔保持邏輯和內容的一致性
- 可追蹤性：文檔間的修復關聯清晰可追蹤

# 使用說明與最佳實踐
# =============================================================================

## 適用場景判斷
- **高適用性**：複雜的 Web 應用、多步驟業務流程、狀態管理複雜的 SPA
- **中適用性**：標準的 CRUD 應用、簡單的 API 調用場景
- **低適用性**：純後端服務、批處理系統、無前端交互的系統

## 執行建議
- **準備充分**：確保相關文檔（FRD、SRD、API 規格）質量良好
- **重點聚焦**：優先分析核心業務流程，避免過度設計
- **團隊協作**：積極與前後端開發團隊溝通，確保設計可行
- **迭代完善**：可分批次完成，優先核心功能的交互設計

## 常見問題處理
- **Q：現有 API 規格不完整怎麼辦？**
  A：先執行 api-specification-generation workflow 完善 API 規格
- **Q：業務流程過於複雜如何處理？**
  A：分解為子流程，分階段設計和實現
- **Q：前後端技術棧差異很大？**
  A：重點關注接口契約，技術實現細節由各自團隊決定

## 與其他 Workflows 的協調
- 建議在 user-story-and-design 完成後、code-implementation 開始前執行
- 可與 api-specification-generation 並行進行，但需要保持一致性
- 執行完成後建議運行 document-consistency-check 確保文檔品質
