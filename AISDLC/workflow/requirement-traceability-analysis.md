# 需求追蹤矩陣分析工作流程 (Requirement Traceability Analysis Workflow)

*此工作流程將用戶的口語化需求描述精確對應到專案文檔中的具體功能，並建立完整的需求追蹤鏈*

---

## Workflow 基本資訊

**名稱**: 需求追蹤矩陣分析工作流程  
**適用場景**: 任何需要將用戶描述對應到專案文檔的開發工作  
**執行頻率**: 每個新開發任務開始前執行  
**預估時間**: 15-30 分鐘  

## 描述

此工作流程解決用戶通常使用口語化、不精確描述開發需求的問題，通過系統性分析將用戶描述對應到專案中的具體功能文檔，建立完整的需求追蹤鏈，確保開發方向準確無誤。

## 適用場景

- **用戶提供口語化需求時機**：當用戶說"我要開發登入功能"、"修改一下付款流程"等模糊描述
- **多功能模糊匹配時機**：當用戶描述可能對應到多個不同的功能模組
- **需求範圍不明確時機**：當需要明確開發邊界和具體範圍
- **跨文檔追蹤時機**：當需要確認 PRD、FRD、SRD、AT 之間的關聯性

## 觸發條件

- ✅ 用戶提供了開發需求描述（無論多麼口語化或模糊）
- ✅ 需要開始任何形式的功能開發工作
- ✅ 專案中已有基本的需求文檔結構（PRD/FRD/SRD）
- ✅ 需要確認開發範圍和優先級

---

## 角色與責任

### 主要負責人 (Primary Owner)
**Agent 角色**: SA Agent (Amanda)  
**責任**: 主導需求分析、建立追蹤矩陣、協調用戶確認過程

### 參與者 (Participants)
- **Dev Agent**: 協助技術可行性評估和開發範圍確認
- **QA Agent**: 協助測試需求分析和測試覆蓋評估
- **PM/PO Agent**: 協助業務優先級確認和功能範圍評估

### 審查者 (Reviewers)
- **用戶/人類決策者**: 確認功能對應正確性、追蹤鏈完整性、開發範圍合理性

---

## 輸入與前置條件

### 必要輸入
- **用戶需求描述**: 用戶提供的任何形式的開發需求描述
- **專案文檔結構**: 專案中現有的 PRD、FRD、SRD、AT 等文檔

### 前置條件
- ✅ 專案中至少存在基本的 PRD 文檔
- ✅ 能夠訪問和搜尋專案文檔
- ✅ 用戶可以參與確認過程
- ✅ SA Agent 具備專案文檔的理解能力

### 所需資源
- **工具和系統**: 文檔搜尋工具、文檔檢視工具
- **時間資源**: 15-30 分鐘的分析和確認時間
- **人力資源**: 用戶參與確認過程

## 執行步驟

## 步驟 1: 用戶需求解析

**描述**: 分析用戶的口語化需求描述，識別關鍵資訊

**執行者**: SA Agent (主導)

**具體作業**:
1. **需求描述收集和記錄**
   - 完整記錄用戶的原始需求描述
   - 識別描述中的關鍵字和概念
   - 注意用戶提到的功能名稱、業務流程、使用場景

2. **關鍵字提取和分析**
   - 提取功能相關的關鍵字（如：登入、註冊、付款、搜尋等）
   - 識別業務流程關鍵字（如：審核、通知、報表等）
   - 分析技術相關詞彙（如：API、介面、資料庫等）

3. **需求意圖推測**
   - 基於關鍵字推測用戶的真實需求意圖
   - 識別可能的功能模組和業務領域
   - 評估需求的複雜度和範圍

4. **潛在歧義識別**
   - 識別描述中模糊或可能有多種理解的部分
   - 標記需要進一步澄清的內容
   - 準備相關的澄清問題

**檢查點**:
- [ ] 用戶原始需求已完整記錄
- [ ] 關鍵字和概念已正確識別
- [ ] 需求意圖推測已完成
- [ ] 潛在歧義已識別並標記

**品質標準**:
- 關鍵字提取準確率 ≥ 95%
- 需求意圖推測必須合理且有依據
- 所有模糊點都必須被識別

---

## 步驟 2: 功能搜尋和對應

**描述**: 在專案文檔中搜尋可能匹配的功能

**執行者**: SA Agent (主導) + Dev Agent (協作)

**具體作業**:
1. **PRD 文檔搜尋**
   - 在所有 PRD 文檔中搜尋相關的功能描述
   - 使用關鍵字進行全文搜尋
   - 識別可能匹配的功能模組和章節

2. **功能匹配度評估**
   - 評估每個找到的功能與用戶需求的匹配程度
   - 分析功能的業務目標是否與用戶需求一致
   - 確認功能的實現範圍是否符合用戶期望

3. **多重匹配處理**
   - 如果找到多個可能的匹配功能，列出所有選項
   - 分析每個選項的差異和特點
   - 準備向用戶展示的比較資訊

4. **無匹配功能處理**
   - 如果沒有找到匹配的功能，確認是否為新功能需求
   - 分析是否需要創建新的需求文檔
   - 評估現有功能是否可以擴展來滿足需求

**檢查點**:
- [ ] PRD 文檔搜尋已完成
- [ ] 所有可能的功能匹配已識別
- [ ] 匹配度評估已完成
- [ ] 多重匹配或無匹配情況已適當處理

**品質標準**:
- 搜尋覆蓋率必須達到 100%
- 匹配度評估必須客觀且有依據
- 所有可能的選項都必須被考慮

---

## 步驟 3: 功能對應確認 🔴 (人類確認點)

**描述**: 向用戶展示功能匹配結果並確認具體要開發的功能

**執行者**: SA Agent (主導) - 與用戶互動

**具體作業**:
1. **匹配結果呈現**
   - 向用戶清晰展示分析結果：
     ```
     🔍 需求分析結果
     
     您的描述: "[用戶原始需求描述]"
     
     我們在專案中找到以下可能的功能匹配：
     
     選項 1: PRD_模組A.md - [功能名稱A]
     ├── 功能描述: [簡要描述功能A的內容]
     ├── 業務目標: [功能A要解決的業務問題]
     └── 實現範圍: [功能A包含的具體功能點]
     
     選項 2: PRD_模組B.md - [功能名稱B] 
     ├── 功能描述: [簡要描述功能B的內容]
     ├── 業務目標: [功能B要解決的業務問題]  
     └── 實現範圍: [功能B包含的具體功能點]
     ```

2. **差異說明**
   - 如有多個選項，詳細說明各選項間的差異
   - 解釋每個選項適合的使用場景
   - 提供選擇建議（如適用）

3. **用戶確認請求**
   - 請用戶明確選擇要開發的具體功能
   - 確認用戶理解所選功能的範圍和內容
   - 記錄用戶的選擇和任何額外說明

4. **新功能需求處理**
   - 如果沒有找到匹配功能，向用戶確認是否為新功能
   - 說明創建新功能文檔的必要性
   - 引導用戶到appropriate的需求創建流程

**檢查點**:
- [ ] 匹配結果已清晰展示給用戶
- [ ] 用戶已明確選擇具體功能
- [ ] 用戶確認已記錄
- [ ] 新功能需求已適當處理

**品質標準**:
- 展示內容必須清晰且易於理解
- 用戶確認必須明確且無歧義
- 所有用戶回饋都必須被記錄

---

## 步驟 4: 需求追蹤鏈建立

**描述**: 基於確認的功能建立完整的文檔追蹤鏈

**執行者**: SA Agent (主導) + Dev Agent + QA Agent (協作)

**具體作業**:
1. **PRD 內容解析**
   - 深入分析確認功能的 PRD 文檔內容
   - 提取業務目標、功能描述、驗收標準
   - 識別相關的利害關係人和約束條件

2. **FRD 文檔追蹤**
   - 搜尋對應的 FRD 文檔
   - 找出相關的 User Stories 和 Acceptance Criteria
   - 分析業務流程和功能交互

3. **SRD 文檔追蹤**
   - 搜尋對應的 SRD 技術文檔
   - 識別相關的技術架構和實現方案
   - 找出相關的 API 規格和資料模型

4. **AT 測試文檔追蹤**
   - 搜尋現有的 AT 測試文檔
   - 評估現有測試的覆蓋率和完整性
   - 識別測試缺口和需要新增的測試案例

5. **追蹤鏈驗證**
   - 確認各文檔之間的關聯性正確
   - 檢查是否有遺漏的文檔或內容
   - 驗證追蹤鏈的完整性

**檢查點**:
- [ ] PRD 內容已完整解析
- [ ] FRD 相關內容已找到並分析
- [ ] SRD 技術內容已找到並分析
- [ ] AT 測試內容已評估
- [ ] 追蹤鏈完整性已驗證

**品質標準**:
- 文檔追蹤覆蓋率必須達到 100%
- 所有關聯關係都必須正確
- 不允許遺漏任何相關文檔

---

## 步驟 5: 追蹤矩陣呈現 🔴 (人類確認點)

**描述**: 向用戶呈現完整的需求追蹤矩陣並確認

**執行者**: SA Agent (主導) - 與用戶互動

**具體作業**:
1. **追蹤矩陣格式化呈現**
   - 使用標準化格式呈現完整的追蹤矩陣：
   
   ```
   📋 需求追蹤矩陣確認
   
   用戶需求: "[用戶原始描述]"
   ↓
   📄 PRD 功能: [PRD 文檔名稱] - [具體功能章節]
   ├── 業務目標: [目標描述]
   ├── 功能描述: [功能詳細描述]
   ├── 利害關係人: [相關角色]
   └── 驗收標準: [關鍵驗收條件]
   
   ↓
   📋 FRD 規格: [FRD 文檔名稱] - [對應章節]  
   ├── User Stories: [US-XXX: 標題, US-YYY: 標題, ...]
   ├── Acceptance Criteria: [AC-XXX-1, AC-XXX-2, AC-YYY-1, ...]
   ├── 業務流程: [主要流程步驟]
   └── 功能依賴: [相關功能模組]
   
   ↓
   ⚙️ SRD 技術規格: [SRD 文檔名稱] - [對應章節]
   ├── 技術架構: [架構模式和設計原則]
   ├── API 規格: [相關 API 端點和規格]
   ├── 資料模型: [相關資料結構和資料庫設計]
   ├── 外部依賴: [第三方服務、資料庫、工具等]
   └── 非功能需求: [效能、安全性、可擴展性要求]
   
   ↓
   🧪 AT 測試案例: [測試文檔狀態]
   ├── 現有測試: [AT-XXX-01: 測試標題, AT-XXX-02: 測試標題, ...]
   ├── 需要新增: [缺失的測試案例類型和數量]
   ├── 測試覆蓋率: [User Story 覆蓋率 X%, AC 覆蓋率 Y%]
   └── 測試優先級: [P0: X個, P1: Y個, P2: Z個]
   ```

2. **關聯性說明**
   - 解釋各文檔之間的關聯關係
   - 說明為什麼這些文檔屬於同一個功能範疇
   - 指出任何特殊的依賴關係或約束

3. **完整性檢查**
   - 向用戶確認追蹤鏈是否包含了所有相關內容
   - 詢問是否有遺漏的功能或相關模組
   - 確認技術實現和測試範圍的合理性

4. **範圍邊界確認**
   - 明確說明此次開發的邊界
   - 識別哪些相關功能不在此次範圍內
   - 確認與其他功能模組的交互點

**檢查點**:
- [ ] 追蹤矩陣已完整呈現
- [ ] 關聯性已清楚說明
- [ ] 用戶已確認追蹤鏈正確性
- [ ] 範圍邊界已明確定義

**品質標準**:
- 追蹤矩陣必須格式化且易於理解
- 所有關聯文檔都必須包含在內
- 用戶確認必須基於充分的資訊

---

## 步驟 6: 開發範圍最終確認 🔴 (人類確認點)

**描述**: 基於追蹤矩陣明確定義開發範圍並獲得最終確認

**執行者**: SA Agent (主導) + PM/PO Agent (協作) - 與用戶互動

**具體作業**:
1. **開發範圍詳細列表**
   - 基於追蹤矩陣製作詳細的開發範圍清單：
   
   ```
   ✅ 本次開發範圍確認
   
   📋 功能範圍:
   ├── 主要功能: [核心功能描述]
   ├── 子功能: [包含的子功能列表]
   └── 邊界功能: [邊界處的功能處理]
   
   📖 User Stories (共 X 個):
   ├── US-XXX: [As a [角色], I want [功能], So that [價值]]
   ├── US-YYY: [As a [角色], I want [功能], So that [價值]]
   └── [其他相關 User Stories...]
   
   ✅ Acceptance Criteria (共 X 個):
   ├── AC-XXX-1: [Given-When-Then 描述]
   ├── AC-XXX-2: [Given-When-Then 描述]  
   ├── AC-YYY-1: [Given-When-Then 描述]
   └── [其他相關 Acceptance Criteria...]
   
   ⚙️ 技術實現範圍:
   ├── 前端開發: [UI/UX 實現範圍]
   ├── 後端開發: [API 和業務邏輯實現]
   ├── 資料庫: [資料模型和資料遷移]
   ├── 第三方整合: [外部服務整合]
   └── 配置和部署: [環境配置需求]
   
   🧪 測試範圍:
   ├── 單元測試: [需要編寫的單元測試]
   ├── 整合測試: [系統整合測試範圍]
   ├── Acceptance Test: [需要執行的 AT 案例]
   └── 效能測試: [效能測試需求，如適用]
   
   📅 預期交付物:
   ├── 功能代碼: [完整的功能實現]
   ├── 測試案例: [完整的測試套件]
   ├── 測試報告: [使用 Test_Report_Template.md]
   ├── 文檔更新: [需要更新的相關文檔]
   └── 部署指南: [部署和配置說明]
   ```

2. **資源和時間評估**
   - 提供大致的開發工作量估算
   - 識別可能的技術挑戰和風險
   - 說明需要的資源和技能要求

3. **優先級確認**
   - 確認各 User Story 和 AC 的優先級
   - 識別核心功能和可選功能
   - 確定最小可行產品 (MVP) 範圍

4. **最終用戶確認**
   - 請用戶確認開發範圍是否符合期望
   - 確認是否有需要調整的地方
   - 獲得最終的開發授權確認

**檢查點**:
- [ ] 開發範圍已詳細列出
- [ ] 資源和時間已評估
- [ ] 優先級已確認
- [ ] 用戶已給予最終確認

**品質標準**:
- 開發範圍必須具體且可執行
- 所有交付物都必須明確定義
- 用戶確認必須完整且明確記錄

## 輸出與交付

### 主要交付物
- **確認的功能對應關係**: 用戶需求與專案功能的精確對應
- **完整的需求追蹤矩陣**: PRD→FRD→SRD→AT 的完整追蹤鏈
- **明確的開發範圍定義**: 詳細的功能、技術、測試範圍清單
- **用戶確認記錄**: 所有確認點的詳細記錄
- **風險和依賴識別**: 開發過程中的潛在風險和依賴關係

### 交付標準
- **追蹤完整性**: 需求追蹤鏈覆蓋率 100%
- **用戶確認度**: 所有確認點都獲得明確的用戶認可
- **範圍清晰度**: 開發範圍邊界明確，無模糊地帶
- **文檔關聯性**: 所有相關文檔都正確關聯且可追蹤

### 驗收條件
- [ ] 用戶需求已精確對應到專案功能
- [ ] 完整的追蹤矩陣已建立並確認
- [ ] 開發範圍已明確定義且獲得用戶確認
- [ ] 所有相關文檔已識別並關聯
- [ ] 潛在風險和依賴已識別

### 後續流程交接
- **交接給**: 各種開發 workflow（如 TDD、需求變更等）
- **交接內容**: 
  - 確認的功能範圍和技術要求
  - 完整的需求追蹤矩陣
  - 用戶確認記錄
- **交接標準**: 
  - 開發範圍明確且無歧義
  - 所有必要的文檔資訊都已準備就緒

## 協作與整合

### 前置 Workflows
- **無特定前置 workflow**: 此為多數開發流程的起始點

### 後續 Workflows  
- **tdd-feature-development-workflow**: 使用此 workflow 的結果進行 TDD 開發
- **requirements-change-management**: 當需求變更時參考原始的追蹤矩陣
- **api-specification-generation**: 基於技術範圍生成 API 規格
- **user-story-and-design**: 使用 User Story 資訊進行設計

### 並行 Workflows
- **無並行 workflow**: 此 workflow 通常需要完成後再進行其他工作

### Agent 協作規則

#### 協作模式
- **SA Agent 主導**: 負責整個需求分析和追蹤過程
- **Dev Agent 支援**: 提供技術可行性評估
- **QA Agent 支援**: 協助測試需求分析
- **PM/PO Agent 顧問**: 提供業務優先級指導

#### 溝通協議
- **用戶互動頻繁**: 多個確認點需要用戶參與
- **即時澄清**: 遇到模糊需求立即向用戶澄清
- **決策透明**: 所有分析過程和建議都要向用戶說明

#### 衝突解決
- **需求解釋衝突**: 
  - SA Agent 提供多種解釋選項
  - 用戶做最終選擇
- **技術可行性爭議**: 
  - Dev Agent 提供技術評估
  - 用戶決定是否接受技術約束或調整需求
- **測試範圍爭議**: 
  - QA Agent 提供測試覆蓋建議
  - 用戶確認測試優先級和範圍

## 品質控制與監控

### 品質檢查點

#### 檢查點 1: 需求解析品質檢查 (步驟 1 完成後)
- **檢查標準**: 用戶需求的關鍵資訊已完整提取且無遺漏
- **檢查方法**: 檢查關鍵字提取的完整性和準確性
- **通過標準**: 關鍵字提取準確率 ≥ 95%，無重要資訊遺漏

#### 檢查點 2: 功能匹配品質檢查 (步驟 2 完成後)
- **檢查標準**: 所有可能的功能匹配都已找到且評估正確
- **檢查方法**: 驗證搜尋覆蓋率和匹配度評估的合理性
- **通過標準**: 搜尋覆蓋率 100%，匹配度評估客觀準確

#### 檢查點 3: 用戶確認品質檢查 (步驟 3 完成後)
- **檢查標準**: 用戶已明確選擇功能且理解所選內容
- **檢查方法**: 確認用戶回饋明確且無歧義
- **通過標準**: 用戶選擇明確，確認記錄完整

#### 檢查點 4: 追蹤鏈品質檢查 (步驟 4 完成後)
- **檢查標準**: 追蹤鏈完整且所有關聯都正確
- **檢查方法**: 驗證 PRD→FRD→SRD→AT 的關聯性和完整性
- **通過標準**: 追蹤覆蓋率 100%，關聯關係正確

#### 檢查點 5: 最終確認品質檢查 (步驟 6 完成後)
- **檢查標準**: 開發範圍明確且獲得完整的用戶確認
- **檢查方法**: 檢查範圍定義的具體性和用戶確認的完整性
- **通過標準**: 範圍定義具體可執行，用戶確認完整明確

### 風險控制

#### 風險 1: 用戶需求理解偏差
- **風險描述**: 對用戶原始需求的理解可能存在偏差
- **緩解措施**: 
  - 多次確認用戶的真實需求意圖
  - 使用具體的功能描述而非抽象概念
  - 提供多個匹配選項讓用戶選擇

#### 風險 2: 功能匹配不準確
- **風險描述**: 找到的功能可能與用戶實際需求不符
- **緩解措施**:
  - 進行全面的文檔搜尋
  - 提供詳細的功能比較說明
  - 讓用戶確認具體的功能內容

#### 風險 3: 追蹤鏈不完整
- **風險描述**: 某些相關文檔或內容可能被遺漏
- **緩解措施**:
  - 系統性地搜尋所有相關文檔類型
  - 建立標準化的追蹤鏈檢查清單
  - 多個 Agent 協作驗證追蹤完整性

#### 風險 4: 範圍蔓延
- **風險描述**: 開發範圍可能在確認過程中不斷擴大
- **緩解措施**:
  - 明確定義範圍邊界
  - 區分核心功能和可選功能
  - 記錄範圍變更並重新確認

### 成功指標

#### 指標 1: 需求對應準確率
- **目標值**: 用戶確認的功能對應準確率 100%
- **測量方法**: 用戶最終確認無需修正的比例
- **監控頻率**: 每次 workflow 執行後

#### 指標 2: 追蹤鏈完整性
- **目標值**: PRD→FRD→SRD→AT 追蹤覆蓋率 100%
- **測量方法**: 檢查追蹤鏈中每個層級的文檔覆蓋情況
- **監控頻率**: 每次追蹤矩陣建立後

#### 指標 3: 用戶確認效率
- **目標值**: 平均確認輪次 ≤ 2 輪
- **測量方法**: 統計從首次呈現到最終確認的互動輪次
- **監控頻率**: 定期統計分析

#### 指標 4: 範圍變更率
- **目標值**: 確認後的範圍變更率 < 10%
- **測量方法**: 統計確認後需要修正範圍的情況
- **監控頻率**: 後續開發完成後回顧

## 特別注意事項

### 用戶互動要求
1. **耐心引導**: 用戶可能不熟悉技術術語，需要用通俗語言解釋
2. **充分說明**: 每個選項都要提供足夠的背景資訊幫助用戶理解
3. **確認記錄**: 所有用戶確認都要詳細記錄，作為後續開發的依據

### 文檔搜尋要求
1. **全面搜尋**: 不能遺漏任何可能相關的文檔
2. **多關鍵字**: 使用多種相關關鍵字進行搜尋
3. **上下文理解**: 不只看關鍵字匹配，還要理解功能的業務上下文

### 追蹤鏈建立要求
1. **關聯驗證**: 確認文檔間的關聯是真實的業務關聯，非偶然關聯
2. **版本一致**: 確保引用的文檔版本是最新且一致的
3. **依賴識別**: 識別功能間的依賴關係和約束條件
