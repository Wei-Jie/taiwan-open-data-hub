基於 AISDLC 框架設計的智能需求提取流程，支持多種輸入形式（截圖、文字、混合），通過強化的人機協作機制確保需求理解準確性。

# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**統一需求提取工作流程 (Unified Requirements Extraction)**

## 描述
此工作流程專門處理各種形式的需求輸入，包括純文字描述、截圖（PPT/Figma）、以及混合輸入。核心特色是"不確定就問，不臆測"原則，所有不確定的推斷都通過人機協作點獲得人類確認，確保需求理解準確性。

## 適用場景
- **輸入類型**：純文字需求、純截圖（1-N張）、截圖+文字說明、混合輸入
- **需求複雜度**：單一功能、複雜流程、多模塊需求
- **確定性程度**：完整描述、部分資訊、模糊需求

## 觸發條件
- 收到任何形式的需求輸入需要分析時
- 需要從視覺材料提取功能需求時
- 多個輸入之間關聯關係不明確時
- 需求資訊不完整需要人類補充時

**Agent 角色**：SA Agent (Amanda - System Analyst)
**责任**：負責輸入解析、需求提取、資訊缺失識別，主导整个人機協作過程

## 参与者 (Participants)
- **BA Agent (Beatrice)**：协助人機協作確認点管理，確保人類輸入得到正确記錄
- **PM/PO Agent (Victoria)**：提供業務优先级和价值判断指导
- **SD Agent (Marcus)**：提供技术可行性初步评估
- **人類使用者**：在所有確認点提供決策，补充缺失資訊，確認理解准确性

## 审查者 (Reviewers)
- **人類使用者**：审查所有需求理解和推断结果
- **BA Agent (Beatrice)**：审查人機協作過程的完整性

# =============================================================================
# 輸入与前置条件
# =============================================================================

## 支持的輸入類型
- **純文字輸入**：使用者需求描述、功能說明、業務流程文字
- **纯截图輸入**：PPT截图、Figma設計稿、原型图、UI界面截图（1-N张）
- **混合輸入**：截图配合文字說明
- **多源輸入**：来自不同渠道的多个輸入项

## 輸入處理要求
- 每个輸入项都需要明确編號和分类
- 缺失的背景資訊需要标记
- 輸入间的关联关系需要識別

## 前置条件
- SA Agent 具备基本的業務领域理解能力
- 人類使用者可配合进行多轮確認和資訊补充
- 有足够时间进行充分的人機協作

# =============================================================================
# 執行步骤
# =============================================================================

## 步驟 1：輸入接收與初步解析
**描述**：接收各种形式的輸入并进行初步分类和編號

**執行者**：SA Agent (主导)

**具体作业**：
1. **輸入分类与編號**
   - 文字輸入：編號为 TXT-001, TXT-002...
   - 截图輸入：編號为 SCR-001, SCR-002...
   - 記錄輸入的原始形式和来源

2. **初步内容識別**
   - 文字内容：提取關鍵词、業務术语、功能描述
   - 截图内容：識別UI元素、頁面類型、操作流程
   - 記錄可識別的明确資訊

3. **輸入完整性评估**
   - 标记資訊完整度（高/中/低）
   - 識別明显缺失的關鍵資訊
   - 准备首轮人機確認

**【人機協作確認点1】輸入理解与完整性確認**：
- **暂停流程**，呈现解析结果
- **呈现内容**：
  - 輸入分类清单：[列出所有輸入及編號]
  - 初步理解：从每个輸入中識別到的内容
  - 关联推测：輸入间可能的关系（如有）
- **人類需要確認**：
  1. 輸入分类是否正确？
  2. 内容理解是否准确？
  3. 这些輸入是否完整？是否需要补充其他資訊？
  4. 輸入间的关联关系是否正确？
- **確認選項**：
  1. 確認理解准确，輸入完整
  2. 修正理解错误，輸入完整
  3. 理解正确，但需补充[具体資訊]
  4. 需要重新分析，提供更多背景

**檢查点**：
- [ ] 所有輸入都已正确分类和編號
- [ ] **人類使用者已確認輸入理解准确性**
- [ ] 需要补充的資訊已明确記錄
- [ ] 輸入间关联关系已获得人類確認

**品质標準**：
- 分类准确且无遗漏
- **人類確認記錄詳細且可追蹤**
- 缺失資訊識別完整

---

## 步骤 2：需求識別与映射分析
**描述**：从各种輸入中識別可能的需求，分析需求间关系

**執行者**：SA Agent (主导)

**具体作业**：
1. **需求提取策略**
   - 单张截图分析：深度識別所有功能点
   - 多张截图分析：判断是否为同一流程或独立功能
   - 文字輸入分析：提取功能需求和業務规则
   - 混合輸入分析：将文字与截图内容匹配

2. **需求映射关系判断**
   - 1輸入→1需求：简单功能
   - 1輸入→N需求：複雜頁面或多功能描述
   - N輸入→1需求：完整流程的多个环节
   - N輸入→N需求：多个独立功能

3. **確定性程度標記**
   - 高確定性：明確可識別的需求
   - 中確定性：需要輕微推斷的需求
   - 低確定性：需要大量假設的需求

**【人機協作確認点2】需求映射与关系確認**：
- **暂停流程**，呈现需求分析结果
- **呈现内容**：
  - 需求提取清单：从每个輸入識別到的需求列表
  - 映射关系分析：[詳細說明輸入与需求的对应关系]
  - 確定性標記：每個需求的可信度級別
  - 推断内容：标记哪些内容是推测的
- **人類需要確認**：
  1. 从截图/文字中識別的需求是否准确？
  2. 需求与輸入的映射关系是否正确？
  3. 遗漏了哪些重要需求？
  4. 哪些推断内容需要修正？
- **確認選項**：
  1. 需求識別准确，映射关系正确
  2. 需求識別正确，但映射关系需調整
  3. 部分需求識別错误，需要修正
  4. 遗漏重要需求，需要补充

**檢查点**：
- [ ] 需求識別覆盖所有輸入内容
- [ ] **人類使用者已確認需求映射准确性**
- [ ] 需求间关系已明确定义
- [ ] 推断内容已获得人類驗證

**品质標準**：
- 需求提取准确且完整
- **映射关系经人類確認无误**
- 確定性標記客觀合理

---

## 步骤 3：資訊缺失識別与补充
**描述**：識別關鍵資訊缺失，主动询问人類补充

**執行者**：SA Agent (主导)

**具体作业**：
1. **關鍵資訊缺失檢查**
   - 使用者角色：谁使用这些功能？
   - 業務流程：完整的操作步骤是什么？
   - 資料来源：資訊从哪里来？
   - 業務规则：有什么限制和约束？
   - 异常處理：出错时如何處理？

2. **假設內容識別**
   - 列出所有基於推測的內容
   - 評估假設對需求理解的影響程度
   - 標記高風險假設（影響重大的推測）

3. **缺失資訊优先级排序**
   - P0：必须了解的核心資訊
   - P1：重要但可暫時假設的資訊
   - P2：有助于完善理解的补充資訊

**【人機協作確認点3】關鍵資訊补充**：
- **暂停流程**，询问缺失的關鍵資訊
- **呈现内容**：
  - 資訊缺失清单：需要补充的關鍵資訊
  - 當前假設內容：標記所有推測部分
  - 影响分析：缺失資訊对需求理解的影响
- **人類需要提供**：
  1. **使用者角色資訊**：[具体询问專案]
  2. **業務流程细节**：[具体询问專案]
  3. **資料和规则**：[具体询问專案]
  4. **异常處理方式**：[具体询问專案]
- **询问格式**：
  ```
  关于 [具体方面]，目前不清楚：
  1. [具体問題1]
  2. [具体問題2]
  
  可能的理解：
  - 選項A：[描述及影响]
  - 選項B：[描述及影响]
  
  请提供准确資訊或选择合适選項
  ```

**檢查点**：
- [ ] 所有關鍵缺失資訊都已識別
- [ ] **人類使用者已提供必要的补充資訊**
- [ ] 高風險假設已通過人類確認或修正
- [ ] 补充資訊已正确整合到需求理解中

**品质標準**：
- 資訊缺失識別全面且准确
- **人類提供的資訊完整記錄**
- 假設內容最小化

---

## 步骤 4：需求整理与初步文档生成
**描述**：基于確認的資訊整理需求，生成初步需求文档

**執行者**：SA Agent (主导)、PM/PO Agent (業務价值)

**具体作业**：
1. **需求分类整理**
   - 功能需求：具体的功能点和操作
   - 非功能需求：性能、安全、可用性等
   - 業務规则：约束条件和處理逻辑
   - 使用者体验需求：界面和交互要求

2. **需求优先级初步排序**
   - 与PM/PO Agent協作评估業務价值
   - 考虑技术實現复杂度
   - 考虑使用者影响程度

3. **初步文档结构生成**
   - 使用AISDLC PRD模板结构
   - 填入已確認的需求内容
   - 标记仍需进一步確認的内容

**【人機協作確認点4】需求整理与优先级確認**：
- **暂停流程**，呈现整理后的需求
- **呈现内容**：
  - 需求分类结果：按类别整理的需求清单
  - 优先级建议：P0/P1/P2分级及理由
  - 初步文档框架：PRD结构和主要内容
  - 待確認專案：仍需进一步明确的内容
- **人類需要確認**：
  1. 需求分类是否合理？
  2. 优先级排序是否符合業務实际？
  3. 文档框架是否完整？
  4. 还需要补充哪些内容？
- **確認選項**：
  1. 確認整理结果，优先级合理
  2. 需求分类合理，但优先级需調整
  3. 分类需要調整，优先级需重新评估
  4. 整理基本正确，但需补充[具体内容]

**檢查点**：
- [ ] 需求已按標準分类整理
- [ ] **人類使用者已確認需求分类和优先级**
- [ ] 初步文档框架已建立
- [ ] 待进一步確認的專案已明确列出

**品质標準**：
- 需求分类逻辑清晰
- **优先级排序经人類確認合理**
- 文档结构符合AISDLC標準

# 品质控制与监控
# =============================================================================

## 品质檢查点
- **步骤1完成后**：檢查輸入解析的准确性和人類確認的完整性
- **步骤3完成后**：驗證關鍵資訊缺失是否已得到有效补充

## 风险控制
- **人類不确定**：提供更詳細的背景資訊和分析，必要时分步骤確認
- **資訊严重缺失**：主动暂停流程，要求提供必要資訊后再继续
- **輸入内容冲突**：明确标识冲突点，要求人類澄清和決策
- **理解分歧**：詳細記錄不同理解可能性，由人類选择正确解读

## 成功指标
- **人類参与度**：100%的關鍵確認点都有人類参与
- **理解准确率**：95%的需求理解在后续阶段无需重大修正  
- **資訊完整度**：90%的關鍵資訊通过人機協作得到补充
- **追蹤完整性**：100%的需求都能追蹤到輸入源和確認過程

# =============================================================================
# 人機協作强化協議
# =============================================================================

## 核心原则
- **不确定就问**：任何不确定的内容都必须询问，禁止自主推测
- **選項明确**：每个询问都提供具体選項和影响分析
- **過程透明**：所有分析過程和確認结果都詳細記錄
- **迭代優化**：基于人類反馈持续改进理解和分析品質

## 主动询问触发条件
- 資訊不完整：缺少關鍵業務資訊
- 逻辑不清晰：存在多种理解可能性
- 内容冲突：輸入间存在矛盾
- 假設過多：需要推測的內容超過30%
- 影响重大：理解错误会显著影响后续開發

## 人類互动標準格式
```
## 情况說明
[詳細描述不确定的内容和分析现状]

## 当前理解
[列出目前的理解和基于的資訊]

## 不确定点
1. [具体不确定的問題1]
2. [具体不确定的問題2]

## 可能選項
1. [選項A] - 影响：[具体影响描述]
2. [選項B] - 影响：[具体影响描述]
3. [其他可能性]

## 建议
[如有明确倾向，說明建议和理由]

## 请確認
请选择正确選項或提供准确資訊：[等待人類輸入]
```

此workflow確保AISDLC框架中的需求提取過程具有最高的准确性和人類监督参与度，为后续流程奠定可靠的基础。
