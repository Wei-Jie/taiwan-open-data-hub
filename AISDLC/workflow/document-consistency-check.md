# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**文檔一致性檢查工作流程 (Document Consistency Check)**

## 描述
此工作流程用於系統性地檢查和驗證 AISDLC 專案中所有文檔的一致性、完整性和正確性。它掃描 PRD、FRD、SRD、API 規格、AT 等所有文檔，檢查需求追蹤鏈的完整性、交叉引用的有效性、內容描述的一致性，並生成詳細的品質報告和改進建議。

## 適用場景
- **定期品質檢查**：每個 Sprint 結束後或里程碑節點進行全面檢查
- **文檔更新後**：當多個文檔同時更新後，確保一致性維護
- **交付前驗證**：重要交付節點前的文檔品質確認
- **新團隊成員加入**：幫助新成員了解文檔現狀和品質要求
- **合規審查準備**：為外部審查或合規檢查做文檔準備

## 觸發條件
- 重要文檔更新完成（如 PRD、FRD 修訂）
- Sprint 週期結束，準備下一階段開發
- 發現文檔間存在不一致的內容
- 新增或修改了需求追蹤關係
- 準備重要交付或里程碑檢查點

# 角色與責任
# =============================================================================

## 主要負責人 (Primary Owner)
**Agent 角色**：SA Agent (Amanda - System Analyst)
**責任**：負責整體文檔一致性檢查的協調、執行和品質報告生成

## 參與者 (Participants)
- **PM/PO Agent (Victoria)**：提供業務需求優先級指導，確認重要不一致問題的修復順序
- **SD Agent (Marcus)**：協助技術文檔（SRD、API規格）的一致性檢查和修復建議
- **QA Agent (Quincy)**：驗證測試相關文檔（AT、Test Report）的一致性和可測試性
- **Backend/Frontend Developer Agent**：提供技術實現角度的一致性驗證
- **人類用戶**：在關鍵決策點確認檢查結果和修復優先級

## 審查者 (Reviewers)
- **人類用戶**：審查一致性檢查報告，確認問題嚴重程度和修復策略
- **PM/PO Agent (Victoria)**：審查業務邏輯相關的不一致問題
- **SD Agent (Marcus)**：審查技術規格相關的不一致問題

# 輸入與前置條件
# =============================================================================

## 必要輸入文檔
- **專案文檔目錄**：完整的專案文檔結構
- **PRD 文檔集**：所有產品需求文檔
- **FRD 文檔集**：所有功能需求文檔
- **SRD 文檔集**：所有系統需求文檔
- **API 規格文檔集**：所有 API 規格文檔（如存在）
- **AT 文檔集**：所有驗收測試文檔
- **變更記錄**：近期的文檔變更歷史

## 前置條件
- 專案文檔基本結構已建立
- 主要文檔（PRD、FRD、SRD）已完成初版
- 文檔目錄結構符合 AISDLC 標準
- 具備讀取所有相關文檔的權限

## 所需資源
- **檢查工具**：文檔掃描和內容分析能力
- **模板標準**：AISDLC 文檔模板規範

- **報告工具**：生成詳細檢查報告的能力

# 執行步驟
# =============================================================================

## 步驟 1：文檔盤點與結構檢查
**描述**：掃描並盤點所有專案文檔，檢查目錄結構和命名規範

**執行者**：SA Agent (主導)

**具體作業**：
1. **目錄結構掃描**
   - 掃描整個專案文檔目錄結構
   - 檢查是否符合 AISDLC 標準結構
   - 識別缺失的必要目錄或文檔
   - 檢查文檔命名是否符合規範

2. **文檔清單建立**
   - 建立完整的文檔清單
   - 記錄每個文檔的基本資訊（檔名、路徑、大小、修改時間）
   - 分類文檔類型（PRD、FRD、SRD、API、AT等）
   - 識別重複、孤立或異常的文檔

3. **必要文檔檢查**
   - 根據專案需求確認必要文檔是否存在
   - 檢查核心文檔的完整性（PRD、FRD、SRD）
   - 驗證 API 相關文檔的存在性（如系統包含 API）
   - 檢查支援文檔的完整性

4. **格式標準初步檢查**
   - 檢查文檔是否使用正確的模板格式
   - 驗證 Markdown 語法的正確性
   - 檢查文檔元資料（標題、版本、日期、作者）

**檢查點**：
- [ ] 專案目錄結構已掃描並符合標準
- [ ] 文檔清單已建立並分類完整
- [ ] 必要文檔存在性已確認
- [ ] 格式標準初步檢查完成

**品質標準**：
- 目錄結構完全符合 AISDLC 標準
- 所有核心文檔類型都有對應文件
- 文檔命名規範統一且清晰

---

## 步驟 2：需求追蹤鏈完整性驗證
**描述**：檢查從 PRD 到 AT 的完整需求追蹤鏈

**執行者**：SA Agent (主導)、QA Agent (協作)

**具體作業**：
1. **編號系統檢查**
   - 檢查 User Story 編號的連續性和唯一性（US-XXX）
   - 驗證 Acceptance Criteria 編號格式（AC-XXX-X）
   - 確認 Acceptance Test 編號對應（AT-XXX-XX）
   - 檢查 API 命名規範的一致性

2. **垂直追蹤鏈驗證**
   - PRD → FRD：確認 PRD 中的需求都有對應的 FRD 描述
   - FRD → SRD：驗證 FRD 中的 User Story 都有對應的技術設計
   - SRD → API規格：檢查 SRD 中的 API 都有詳細規格文檔
   - FRD → AT：確認 FRD 中的 AC 都有對應的測試案例

3. **水平關聯檢查**
   - 檢查同一需求在不同文檔中的一致性
   - 驗證相關需求之間的依賴關係
   - 確認需求變更的影響範圍追蹤

4. **追蹤矩陣更新**
   - 建立或更新需求追蹤矩陣
   - 標識缺失的追蹤關係
   - 記錄追蹤鏈斷裂點

**【人機協作確認點1】追蹤鏈問題確認**：
- **暫停流程**，呈現追蹤鏈檢查結果
- **呈現內容**：
  - 追蹤鏈完整性報告：完整和缺失的追蹤關係
  - 編號系統問題：重複、缺失或格式錯誤的編號
  - 斷裂點分析：具體的追蹤鏈斷裂位置和影響
  - 修復建議：針對每個問題的具體修復方案
- **人類需要確認**：
  1. 追蹤鏈斷裂的嚴重程度評估？
  2. 哪些問題需要優先修復？
  3. 是否接受建議的修復方案？
  4. 是否有其他需要特別關注的追蹤關係？
- **確認選項**：
  1. 確認問題評估，按建議優先級修復
  2. 調整問題優先級，重新排序修復順序
  3. 某些問題暫時接受，專注於高優先級問題
  4. 需要更詳細的影響分析再決定

**檢查點**：
- [ ] 需求編號系統檢查完成
- [ ] **人類用戶已確認追蹤鏈問題優先級**
- [ ] 垂直追蹤鏈完整性已驗證
- [ ] 追蹤矩陣已更新

**品質標準**：
- 需求追蹤鏈 95% 以上完整
- **追蹤問題經人類確認並有修復計劃**
- 編號系統統一且無重複

---

## 步驟 3：交叉引用完整性檢查
**描述**：檢查所有文檔間的內部連結和交叉引用

**執行者**：SA Agent (主導)

**具體作業**：
1. **內部連結掃描**
   - 掃描所有 Markdown 連結語法
   - 檢查相對路徑和絕對路徑的正確性
   - 驗證錨點連結（#section）的有效性
   - 識別所有外部連結

2. **交叉引用驗證**
   - 檢查文檔間引用的準確性
   - 驗證引用的章節或段落是否存在
   - 確認引用內容與被引用內容的一致性
   - 檢查圖片、表格等資源的連結

3. **斷裂連結檢測**
   - 識別所有無效的內部連結
   - 檢測指向不存在文件或章節的連結
   - 找出路徑錯誤或檔名變更導致的斷裂
   - 記錄需要修復的連結清單

4. **孤立文檔識別**
   - 找出沒有被其他文檔引用的文件
   - 識別可能遺忘的重要文檔
   - 檢查是否需要建立連結關係

**檢查點**：
- [ ] 所有內部連結已掃描驗證
- [ ] 交叉引用準確性已檢查
- [ ] 斷裂連結清單已建立
- [ ] 孤立文檔已識別

**品質標準**：
- 內部連結有效率 98% 以上
- 所有重要文檔都有適當的交叉引用
- 無孤立的核心文檔

---

## 步驟 4：內容一致性深度檢查
**描述**：檢查相同概念在不同文檔中的描述一致性

**執行者**：SA Agent (主導)、SD Agent (技術內容)、PM/PO Agent (業務內容)

**具體作業**：
1. **業務概念一致性**
   - 檢查同一業務需求在 PRD、FRD 中的描述一致性
   - 驗證業務術語使用的統一性
   - 確認業務流程描述的一致性
   - 檢查使用者角色定義的一致性

2. **技術規格一致性**
   - 比對 FRD 和 SRD 中技術需求的一致性
   - 檢查 API 規格與 SRD 設計的對應
   - 驗證資料庫設計在不同文檔中的一致性
   - 確認系統架構描述的統一性

3. **測試內容一致性**
   - 檢查 AT 與 FRD 中 AC 的對應關係
   - 驗證測試案例與需求的一致性
   - 確認測試資料與業務場景的匹配
   - 檢查測試結果與預期行為的一致性

4. **版本資訊一致性**
   - 檢查相關文檔的版本號同步
   - 驗證修訂歷史的一致性
   - 確認變更記錄的完整性
   - 檢查文檔日期的邏輯性

**檢查點**：
- [ ] 業務概念一致性已檢查
- [ ] 技術規格一致性已驗證
- [ ] 測試內容一致性已確認
- [ ] 版本資訊一致性已檢查

**品質標準**：
- 核心概念描述 90% 以上一致
- 技術規格無重大矛盾
- 版本資訊完全同步

---

## 步驟 5：格式標準深度驗證
**描述**：深度檢查所有文檔的格式標準符合度

**執行者**：SA Agent (主導)

**具體作業**：
1. **模板符合度檢查**
   - 對比每個文檔與對應模板的結構
   - 檢查必填欄位的完整性
   - 驗證章節標題和編號的正確性
   - 確認表格格式的統一性

2. **Markdown 語法驗證**
   - 檢查標題層級的邏輯性
   - 驗證清單格式的一致性
   - 確認程式碼區塊的正確性
   - 檢查表格語法的規範性

3. **文檔元資料驗證**
   - 檢查文件資訊表格的完整性
   - 驗證修訂歷史的格式統一性
   - 確認作者資訊的準確性
   - 檢查日期格式的一致性

4. **內容結構一致性**
   - 檢查同類型文檔結構的統一性
   - 驗證章節編號系統的一致性
   - 確認引用格式的標準化
   - 檢查專有名詞的一致性

**檢查點**：
- [ ] 模板符合度已全面檢查
- [ ] Markdown 語法已驗證
- [ ] 文檔元資料已確認
- [ ] 內容結構一致性已檢查

**品質標準**：
- 模板符合度 95% 以上
- Markdown 語法完全正確
- 文檔元資料完整且準確

---

## 步驟 6：綜合報告生成與修復建議
**描述**：生成完整的一致性檢查報告和修復建議

**執行者**：SA Agent (主導)

**具體作業**：
1. **問題分類與優先級排序**
   - 將所有發現的問題分類（致命、嚴重、一般、輕微）
   - 根據業務影響和修復難度排序
   - 建立問題修復的時間線
   - 標識需要協作修復的問題

2. **修復建議制定**
   - 為每個問題提供具體的修復方案
   - 估算修復所需的時間和資源
   - 識別修復過程中的風險和依賴
   - 提供最佳實踐建議

3. **品質評分與趨勢分析**
   - 對整體文檔品質進行量化評分
   - 與歷史檢查結果比較趨勢
   - 識別持續改進的機會
   - 建立品質監控機制

4. **後續行動計劃**
   - 制定詳細的修復執行計劃
   - 分配修復責任和時間節點
   - 建立修復進度追蹤機制
   - 規劃下次檢查的時間和範圍

**【人機協作確認點2】檢查結果和修復策略確認**：
- **暫停流程**，呈現完整的檢查報告
- **呈現內容**：
  - 一致性檢查總報告：所有問題的綜合分析
  - 問題優先級排序：按嚴重程度和影響範圍分類
  - 修復建議清單：每個問題的具體修復方案
  - 品質評分報告：整體文檔品質的量化評估
  - 行動計劃：建議的修復時間線和責任分配
- **人類需要確認**：
  1. 是否同意問題的優先級排序？
  2. 修復建議是否可行且合理？
  3. 時間線和資源分配是否合適？
  4. 是否需要調整修復策略？
- **確認選項**：
  1. 確認所有建議，按計劃執行修復
  2. 調整部分優先級和時間線
  3. 某些問題延後處理，專注核心問題
  4. 需要更詳細的成本效益分析

**檢查點**：
- [ ] 所有問題已分類和排序
- [ ] **人類用戶已確認修復策略**
- [ ] 修復建議已制定
- [ ] 後續行動計劃已建立

**品質標準**：
- 問題識別準確且全面
- **修復建議經人類確認可行**
- 行動計劃具有可操作性

- **所有核心 workflows**：需要有基本的文檔產出才能進行一致性檢查
- **requirements-change-management**：變更管理後需要檢查文檔一致性

## 後續 Workflows  
- **特定修復 workflows**：根據發現的問題觸發相應的修復流程
- **定期維護 workflows**：建立定期檢查機制

## 並行 Workflows
*無特定並行 workflow 需求，但可與其他維護活動同時進行*

## Agent 協作規則
- **協調模式**：SA Agent 主導整體流程，其他 Agent 提供專業領域的檢查支持
- **專業分工**：SD Agent 負責技術內容，PM/PO Agent 負責業務內容，QA Agent 負責測試內容
- **人機協作協議**：關鍵決策點需要人類確認，確保檢查結果的準確性和修復策略的合理性

# 品質控制與監控
# =============================================================================

## 品質檢查點
- **初步檢查後**：驗證檢查範圍和方法的正確性
- **問題識別後**：確認問題分類和優先級的準確性
- **報告生成前**：進行最終的品質驗證

## 風險控制
- **檢查遺漏風險**：建立系統性的檢查清單，確保全面覆蓋
- **誤報問題風險**：多重驗證機制，避免錯誤的問題識別
- **修復建議不當**：與相關專家協作，確保建議的可行性
- **時間壓力風險**：合理安排檢查時間，避免匆忙執行影響品質

## 成功指標
- **檢查覆蓋率**：100% 的專案文檔都納入檢查範圍
- **問題識別準確率**：≥ 95% 的識別問題確實存在
- **修復建議採納率**：≥ 90% 的修復建議被接受並執行
- **文檔品質改進率**：檢查後文檔品質分數顯著提升
- **團隊滿意度**：對檢查結果和修復建議的滿意度 ≥ 4.0/5.0

此 workflow 確保 AISDLC 框架中的所有文檔保持高度一致性和標準化，為軟體開發提供可靠的文檔基礎，並建立持續改進的品質管理機制。
