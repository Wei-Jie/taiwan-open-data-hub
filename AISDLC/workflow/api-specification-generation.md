# =============================================================================
# Workflow 基本資訊
# =============================================================================

## Workflow 名稱
**API 規格生成與更新工作流程 (API Specification Generation and Update)**

## 描述
此工作流程用於系統性地檢查、生成和更新 API 規格文檔。它掃描現有的 SRD 文檔以識別 API 定義，從相關的 User Story、Acceptance Criteria 和 Acceptance Test 中提取需求資訊，為每個 API 生成獨立的規格文件，並建立與源文檔的雙向關聯。確保 API 文檔的完整性、準確性和可追蹤性。

## 適用場景
- **系統設計完成後**：SRD 文檔中包含 API 定義時
- **API 規格缺失**：發現 API 沒有對應的詳細規格文檔
- **需求變更影響 API**：User Story 或系統需求變更導致 API 介面修改
- **定期維護**：確保 API 文檔與實際需求保持同步

## 觸發條件
- SRD 文檔中新增或修改 API 定義
- User Story 包含 API 相關需求但缺少詳細規格
- API 規格文件不存在或內容不完整
- 需求變更管理流程中識別出 API 介面變更

# =============================================================================
# 角色與責任
# =============================================================================

## 主要負責人 (Primary Owner)
**Agent 角色**：SD Agent (Marcus - System Designer)
**責任**：負責 API 規格的技術準確性、文檔格式標準化、與 SRD 的一致性維護

## 參與者 (Participants)
- **SA Agent (Amanda)**：提供業務需求解讀、協助 User Story 到 API 的映射
- **Backend Developer Agent**：提供 API 技術實現細節、參數驗證規則
- **QA Agent (Quincy)**：提供測試案例、驗證 API 規格的可測試性
- **人類用戶**：在關鍵決策點確認 API 規格與業務需求的一致性

## 審查者 (Reviewers)
- **SA Agent (Amanda)**：審查 API 規格與 User Story 的一致性
- **Backend Developer Agent**：審查 API 技術規格的合理性和可實現性
- **人類用戶**：最終審查 API 規格的業務邏輯正確性

# =============================================================================
# 輸入與前置條件
# =============================================================================

## 必要輸入文檔
- **SRD 文檔**：包含 API 設計章節的系統需求文檔
- **FRD 文檔**：包含相關 User Story 和 Acceptance Criteria
- **AT 文檔**：包含 API 相關的驗收測試案例
- **現有 API 規格文件**（如存在）：需要更新的現有 API 文檔

## 前置條件
- SRD 文檔已完成並包含 API 設計章節
- 相關的 FRD 和 AT 文檔已完成並可供參考
- API_Specification_Template.md 模板可用
- 項目文檔目錄結構已建立

## 所需資源
- **模板資源**：API 規格模板、API 索引模板
- **分析工具**：文檔掃描和內容提取能力
- **時間資源**：每個 API 約 30-60 分鐘的規格撰寫時間
- **驗證環境**：用於驗證 API 規格的技術可行性

# =============================================================================
# 執行步驟
# =============================================================================

## 步驟 1：API 盤點與現狀分析
**描述**：掃描現有文檔，識別所有 API 定義並評估現有規格文件的完備性

**執行者**：SD Agent (主導)

**具體作業**：
1. **掃描 SRD 文檔識別 API**
   - 解析所有 SRD 文檔中的「API 設計」章節
   - 提取 API 端點、HTTP 方法、基本描述
   - 建立 API 清單，包含模組名稱和端點資訊

2. **檢查現有 API 規格文件**
   - 掃描 srd/api/ 目錄下的現有 API 文檔
   - 評估每個 API 文件的完整性（基於模板檢查清單）
   - 識別缺失或不完整的 API 規格文件

3. **建立 API 狀態報告**
   - 分類：完整、不完整、缺失
   - 優先級排序：基於 User Story 重要性和開發排程
   - 工作量估算：需要生成或更新的工作量

**檢查點**：
- [ ] 所有 SRD 文檔已掃描並提取 API 清單
- [ ] 現有 API 規格文件完整性已評估
- [ ] API 狀態報告已生成並包含優先級排序

**品質標準**：
- API 清單涵蓋所有 SRD 中定義的端點
- 完整性評估基於標準模板的所有必填項目
- 優先級排序考慮業務價值和技術依賴

---

## 步驟 2：需求資訊收集與分析
**描述**：從相關的 User Story、AC 和 AT 中收集 API 需求資訊

**執行者**：SA Agent (主導)、SD Agent (協作)

**具體作業**：
1. **User Story 分析**
   - 識別與每個 API 相關的 User Story
   - 提取業務場景、用戶角色、期望行為
   - 確定 API 的業務價值和使用場景

2. **Acceptance Criteria 解析**
   - 提取 API 相關的驗收標準
   - 識別必要的請求參數和驗證規則
   - 確定成功和失敗的回應條件

3. **Acceptance Test 分析**
   - 收集 API 相關的測試案例
   - 提取請求範例和預期回應
   - 識別邊界條件和錯誤處理場景

4. **需求資訊結構化**
   - 按 API 分組整理收集到的資訊
   - 建立需求到 API 參數的映射關係
   - 識別可能的資訊缺口和需要澄清的部分

**【人機協作確認點1】需求資訊確認**：
- **暫停流程**，呈現收集到的需求資訊
- **呈現內容**：
  - API 需求映射表：每個 API 對應的 User Story 和 AC
  - 業務場景分析：API 的使用場景和業務邏輯
  - 參數需求清單：從需求中提取的 API 參數要求
  - 資訊缺口報告：需要進一步澄清的部分
- **人類需要確認**：
  1. API 與 User Story 的映射關係是否正確？
  2. 業務場景理解是否準確？
  3. API 參數需求是否完整？
  4. 是否有遺漏的重要需求？
- **確認選項**：
  1. 確認需求資訊收集完整準確
  2. 需要補充特定 API 的業務邏輯說明
  3. 需要澄清某些 User Story 與 API 的關聯
  4. 發現重要需求遺漏，需要重新分析

**檢查點**：
- [ ] 所有 API 相關的 User Story 已識別
- [ ] **人類用戶已確認需求資訊收集準確性**
- [ ] Acceptance Criteria 中的 API 要求已提取
- [ ] Acceptance Test 案例已收集並分析

**品質標準**：
- 每個 API 都有對應的業務需求依據
- **需求資訊經人類確認與實際期望一致**
- 參數需求具有明確的業務邏輯支持

---

## 步驟 3：API 規格文件生成
**描述**：基於 API_Specification_Template.md 為每個 API 生成獨立的規格文件

**執行者**：SD Agent (主導)

**具體作業**：
1. **文件結構準備**
   - 在 srd/api/ 目錄下準備文件結構
   - 按命名規範創建 API 文件：API_[模組]_[端點名稱].md
   - 確保文件命名與 API 功能和模組的對應關係清晰

2. **基本資訊填寫**
   - 使用模板創建新的 API 規格文件
   - 填寫文件資訊、修訂歷史、API 摘要
   - 設定環境資訊和基礎路徑

3. **API 詳細規格撰寫**
   - 填寫端點資訊：HTTP 方法、路徑、描述
   - 定義請求參數：基於需求分析結果
   - 設計回應格式：包含成功和錯誤情況
   - 添加請求/回應範例

4. **認證與權限設定**
   - 基於 User Story 中的角色定義權限要求
   - 設定認證機制和授權範圍
   - 定義安全相關的參數和標頭

5. **關聯文檔連結**
   - 添加「關聯文檔」章節
   - 連結到相關的 SRD、FRD、AT 文檔
   - 建立需求追蹤鏈

**檢查點**：
- [ ] 所有 API 文件已按命名規範創建
- [ ] API 規格內容完整且符合模板格式
- [ ] 請求參數和回應格式基於需求分析定義
- [ ] 關聯文檔連結已建立

**品質標準**：
- API 文件嚴格遵循 API_Specification_Template.md 格式
- 所有必填欄位已完整填寫
- 參數定義具有明確的業務邏輯依據

---

## 步驟 4：關聯建立與索引更新
**描述**：建立 API 文件與 SRD 的雙向關聯，更新 API 索引

**執行者**：SD Agent (主導)

**具體作業**：
1. **更新 SRD 文檔**
   - 在 SRD 的「API 設計」章節中添加詳細規格連結
   - 將原有的簡要 API 描述替換為規格文件連結
   - 確保 SRD 與 API 規格文件的一致性

2. **創建 API 索引文件**
   - 創建 srd/api/API_Index.md 文件
   - 列出所有 API 的概覽資訊
   - 提供按模組和功能分類的導航

3. **建立交叉引用**
   - 在每個 API 文件中添加相關 API 的交叉引用
   - 識別 API 間的依賴關係
   - 建立 API 組合使用的說明

4. **驗證連結完整性**
   - 檢查所有內部連結的有效性
   - 確認文檔間的引用關係正確
   - 驗證追蹤鏈的完整性

**檢查點**：
- [ ] SRD 文檔已更新並包含 API 規格連結
- [ ] API_Index.md 已創建並包含所有 API 概覽
- [ ] 文檔間的交叉引用已建立
- [ ] 所有連結已驗證有效

**品質標準**：
- SRD 與 API 規格文件內容保持一致
- API 索引提供清晰的導航結構
- 追蹤鏈完整且可驗證

---

## 步驟 5：品質驗證與最終確認
**描述**：對生成的 API 規格進行全面品質檢查

**執行者**：SD Agent (主導)、QA Agent (品質檢查)

**具體作業**：
1. **格式標準驗證**
   - 檢查所有 API 文件是否符合模板格式
   - 驗證必填欄位的完整性
   - 確認 Markdown 語法和結構正確性

2. **內容一致性檢查**
   - 驗證 API 規格與 SRD 的一致性
   - 檢查參數定義與 User Story 需求的對應
   - 確認測試案例與 AT 文檔的一致性

3. **技術可行性驗證**
   - 評估 API 設計的技術合理性
   - 檢查參數類型和約束的正確性
   - 驗證錯誤處理機制的完整性

4. **可測試性評估**
   - 與 QA Agent 協作評估 API 規格的可測試性
   - 確認測試案例涵蓋所有場景
   - 驗證 API 規格提供足夠的測試依據

**【人機協作確認點2】API 規格最終確認**：
- **暫停流程**，呈現完整的 API 規格文檔
- **呈現內容**：
  - API 規格文件清單：所有生成的 API 文檔
  - 品質檢查報告：格式、內容、技術驗證結果
  - 關聯關係圖：API 文檔與需求文檔的連結關係
  - 覆蓋率分析：API 規格對需求的覆蓋程度
- **人類需要確認**：
  1. API 規格內容是否準確反映業務需求？
  2. 技術規格是否合理且可實現？
  3. 文檔品質是否達到標準？
  4. 是否需要調整或補充任何內容？
- **確認選項**：
  1. 確認所有 API 規格完整正確
  2. 需要調整特定 API 的技術細節
  3. 需要補充某些 API 的業務邏輯說明
  4. 發現重大問題，需要重新生成部分規格

**檢查點**：
- [ ] 所有 API 文件格式符合標準模板
- [ ] **人類用戶已確認 API 規格準確性**
- [ ] 內容一致性檢查通過
- [ ] 技術可行性和可測試性已驗證

**品質標準**：
- 所有 API 規格文檔格式標準且內容完整
- **API 規格經人類確認符合業務需求**
- 技術規格合理且具有可實現性
