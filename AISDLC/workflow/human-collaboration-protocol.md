# 人機協作協議

AISDLC 框架中 Agent 與人類用戶協作的標準化協議，確保高質量的人機交互和決策過程。

# =============================================================================
# 協議概述
# =============================================================================

## 協議目的
定義 AISDLC 框架中 Agent 與人類用戶協作的標準化規範，確保：
- 關鍵決策點都有人類參與和確認
- 人機交互過程標準化、可追蹤
- Agent 在不確定情況下主動尋求人類指導
- 所有協作過程都有完整記錄

## 核心原則
- **🚫 絕對禁止推測**：Agent 遇到任何不確定情況必須立即停止，禁止任何形式的自主推測或"合理"假設
- **⏱️ 硬停止機制**：設置強制等待時間限制，超時必須升級處理
- **🔒 多重驗證要求**：關鍵決策必須通過多重確認才能繼續
- **👁️ 透明化過程**：所有分析過程和推理邏輯都向人類透明展示
- **📊 選項化決策**：為人類提供明確的選項和影響分析，便於決策
- **📝 完整記錄**：所有人機交互都詳細記錄，確保可追蹤性
- **🔄 迭代優化**：基於人類反饋持續優化協作質量
- **🆘 強制升級機制**：複雜或關鍵問題必須升級至更高權限人員

## 適用範圍
- 所有 AISDLC workflow 中的人機協作確認點
- Agent 遇到不確定情況需要人類指導時
- 關鍵業務決策和技術方案選擇時
- 需求理解和驗證過程中的澄清確認

# =============================================================================
# 人機協作確認點類型
# =============================================================================

## 🔴 強制確認點 (Mandatory Confirmation) - 硬約束級別

**觸發條件**：
- 資訊不完整，缺少關鍵業務資訊
- 存在多種理解可能性，需要人類澄清  
- 輸入內容之間存在矛盾或衝突
- 重要的業務決策和優先級確認
- **任何需要假設或推測的情況**
- **規格解釋存在歧義的情況**
- **業務邏輯不明確的情況**

**硬約束處理要求**：
- **⛔ 立即停止**：必須立即暫停 workflow 執行，不允許任何形式的繼續
- **⏰ 30分鐘硬限制**：如人類 30 分鐘內未回應，自動觸發升級機制
- **🔒 零推測原則**：絕不允許"基於常識"、"按照慣例"或"合理推斷"等方式自行決定
- **📋 強制記錄**：所有停止原因、等待時間、升級過程都必須詳細記錄
- **🆘 自動升級**：超時後自動通知更高權限人員或專家
- **🔍 多重驗證**：恢復執行前必須經過至少 2 個 Agent 的一致性確認

**升級觸發條件**：
- 人類 30 分鐘內未回應
- 人類回應不明確或仍存在歧義
- 涉及關鍵業務邏輯或技術架構決策
- 同一類問題反覆出現 3 次以上

## 🟡 建議確認點 (Recommended Confirmation)
**觸發條件**：
- 推斷內容較多，確認可提高準確性
- 業務流程或技術方案的合理性驗證
- 工作量估算和資源規劃的確認

**處理要求**：
- 建議進行人類確認
- 如人類不可用，可基於最佳實務繼續
- 記錄推斷內容和決策依據

## 🟢 資訊確認點 (Information Confirmation)
**觸發條件**：
- 格式規範和命名標準的應用
- 模板使用和文檔結構的確認
- 流程進度和下一步驟的通知

**處理要求**：
- 提供資訊供人類了解
- 不需要暫停等待確認
- 保持資訊記錄

### 實際應用範例
```markdown
## 【人機協作確認點1】輸入理解與完整性確認

### 情況說明
已收到3張截圖和2段文字描述，需要確認對輸入內容的理解是否準確，以及是否需要補充其他資訊。

### 當前理解/分析結果
從提供的材料中識別到：
- SCR-001: 登录頁面截图，包含用户名、密码输入框
- SCR-002: 主頁面列表截圖，顯示資料表格和操作按鈕
- SCR-003: 詳情頁面截圖，展示單項資料的詳細資訊
- TXT-001: "需要支持多种用户角色的权限管理"
- TXT-002: "系統需要支援資料匯出功能"

### 呈現內容
- **輸入分類清單**：3張截圖(登錄、列表、詳情) + 2段功能描述
- **初步理解**：用戶認證 + 資料管理 + 權限控制 + 資料匯出
- **關聯推測**：截圖展示完整的使用者操作流程，文字補充功能需求

### 人類需要確認
1. 對截圖內容的理解是否準確？
2. 截圖間的流程關係理解是否正確？
3. 文字描述與截圖的關聯是否合理？
4. 是否需要補充其他重要資訊？

### 確認選項
1. 確認理解準確，輸入完整 - 影響：直接進入需求分析
2. 理解基本正確，需補充[具體資訊] - 影響：補充後繼續分析
3. 截圖理解有誤，需要糾正 - 影響：重新分析相關內容
4. 需要更多背景資訊才能準確理解 - 影響：暫停等待補充

### 建議
建議選擇選項1或2，當前理解基本準確，如有補充資訊可進一步完善。

### 請確認
請選擇合適的選項或提供準確資訊：[等待人類輸入]
```

# 主動詢問機制
# =============================================================================

## 觸發主動詢問的情況

### 資訊不完整 (Information Incomplete)
```yaml
觸發條件:
  - 缺少關鍵業務資訊（用戶角色、業務流程、資料來源等）
  - 輸入材料不足以支撐完整的需求理解
  - 關鍵假設過多（超過30%的內容需要推測）

詢問策略:
  - 明確列出缺失的資訊類型
  - 提供資訊重要性的優先級排序
  - 說明缺失資訊對後續分析的影響
```

### 邏輯不清晰 (Logic Unclear)
```yaml
觸發條件:
  - 存在多種合理的理解可能性
  - 業務流程邏輯不明確
  - 需求間存在潛在衝突

詢問策略:
  - 列舉所有可能的理解方式
  - 分析每種理解的合理性和影響
  - 請求人類澄清正確的理解
```

### 內容衝突 (Content Conflict)
```yaml
觸發條件:
  - 不同輸入材料之間存在矛盾
  - 功能需求與業務約束衝突
  - 用戶描述與截圖內容不一致

詢問策略:
  - 明確指出衝突的具體內容
  - 分析可能的解決方案
  - 請求人類決定正確的處理方式
```

### 影响重大 (High Impact)
```yaml
觸發條件:
  - 理解錯誤會顯著影響項目進度或成本
  - 涉及核心業務邏輯的關鍵決策
  - 技術方案選擇對系統架構有重大影響

詢問策略:
  - 詳細說明決策的重要性和影響
  - 提供充分的分析和建議
  - 確保人類充分理解後再做決策
```

## 🆕 硬約束機制執行方案

### 強制停止執行機制
```yaml
觸發場景: 遇到強制確認點
硬約束執行:
  1. ⛔ 立即停止所有推理和決策過程
  2. 🚨 顯示紅色警示：**工作流程已暫停，等待人類確認**
  3. ⏰ 啟動 30 分鐘倒計時機制
  4. 📋 記錄停止時間、原因、當前狀態
  5. 🔄 持續顯示等待狀態，不允許任何自主繼續

硬限制違規檢測:
  - 自動檢測是否有任何形式的自主推測
  - 監控是否有跳過確認點的行為
  - 驗證是否有未經確認的假設被使用
```

### 30分鐘超時升級機制
```yaml
觸發條件: 人類 30 分鐘內未回應強制確認點
自動執行流程:
  1. 🆘 觸發升級警報
  2. 📧 自動通知指定的升級聯繫人
  3. 📝 生成詳細的升級報告
  4. ⏸️ 維持工作流程暫停狀態
  5. 🔍 等待更高權限人員的指示

升級報告內容:
  - 當前工作流程和確認點詳情
  - 需要澄清的具體問題
  - 可能的選項和影響分析
  - 等待時間和嘗試次數
  - 建議的處理方案
```

### 人類回應品質驗證機制
```yaml
觸發場景: 收到人類回應後
品質驗證步驟:
  1. 🔍 檢查回應是否完整回答所有確認問題
  2. 📊 驗證選擇的選項是否明確
  3. 🤔 確認是否仍存在歧義或不確定點
  4. 🔄 如不完整，禁止繼續並要求澄清
  5. ✅ 通過驗證後才允許恢復工作流程

不合格回應處理:
  - 禁止基於不完整回應進行推測
  - 要求提供更具體的澄清
  - 重新設計問題使其更清晰
  - 超過 3 次不合格自動升級
```

### AI 幻覺檢測與防護機制
```yaml
實時監控項目:
  1. 🚫 檢測任何形式的自主推測行為
  2. 🔍 監控是否有未經確認的假設
  3. 📋 驗證是否所有規格點都有明確來源
  4. 🎯 檢查決策是否基於確認的資訊

防護措施:
  - 自動標記所有假設和推測
  - 強制要求標明資訊來源
  - 禁止使用"常見做法"、"一般來說"等模糊表述
  - 要求對每個決策提供具體的確認依據
```

## 常見問題及解決方案（強化版）

### 人類不可用時的處理（強化版）
```yaml
情況: 人類暫時無法參與確認
硬約束處理:
  1. ⛔ 對於強制確認點，必須立即暫停，絕不繼續
  2. ⏰ 啟動 30 分鐘硬限制倒計時
  3. 📞 建立緊急聯繫和升級機制
  4. 🚨 顯示明確的暫停狀態和原因
  5. 📝 記錄所有等待時間和升級過程

禁止行為:
  - 基於"最佳實務"自行決定
  - 採用"常見做法"繼續工作
  - 做任何形式的"合理推斷"
```

### 人類回應不明確時的處理（強化版）
```yaml
情況: 人類的回應模糊或不完整
品質驗證處理:
  1. 🔍 立即檢測回應完整性
  2. 🚫 禁止基於不完整回應進行推測
  3. 📋 列出具體需要澄清的問題
  4. 🔄 重新設計更清晰的確認選項
  5. ⏰ 重新啟動確認流程和計時

升級條件:
  - 連續 3 次回應不明確
  - 澄清後仍存在關鍵歧義
  - 涉及核心業務邏輯的模糊回應
```

### 反覆確認同類問題的處理（強化版）
```yaml
情況: 類似問題需要多次確認
AI 幻覺風險評估:
```

## 協作效果評估

### 量化指標
- **確認準確率**：人類確認後在後續階段無需重大修正的比例

### 定性評估
