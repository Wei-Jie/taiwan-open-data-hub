# AI 協作框架快速參考指南

> **最後更新**：2025-11-09
> **目的**：提供 AI 輔助開發框架的快速對比，幫助選擇適合的工具組合

---

## 📊 四框架定位矩陣

| 框架 | 核心定位 | 比喻 | 主要解決的問題 |
|------|---------|------|---------------|
| **Spec-Kit** | AI 開發工具鏈 | 萬用電動工具箱 (DeWalt, Makita) | 如何**與 AI 高效互動** |
| **OpenSpec** | 規格狀態管理系統 | Git 版本控制系統 | 如何**原子化管理變更** |
| **AISDLC** | AI 開發生命週期框架 | 精密工廠生產線（豐田生產系統） | 如何**組織 AI 協作與治理** |
| **BMAD-METHOD** | 適應性協作框架 | 靈活的專家顧問團隊（McKinsey） | 如何**最大化靈活性與協作深度** |

## 🎯 使用場景對比

| 場景 | Spec-Kit | OpenSpec | AISDLC | BMAD | 推薦組合 | 推薦原因 |
|------|---------|---------|--------|------|---------|---------|
| **個人小型專案** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | Spec-Kit + OpenSpec | Spec-Kit 提供快速的 CLI 工具和開發環境（Dev Container），OpenSpec 提供清晰的變更管理。這個組合既高效又不過度複雜。BMAD 的 Quick Flow 模式可用但整體框架對小專案略重，AISDLC 的完整流程對小專案而言過重。 |
| **企業級專案** | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | AISDLC 2.0（整合四者） | 企業級專案需要完整的角色定義、流程規範和品質保證（AISDLC 治理層），清晰的狀態與變更管理（OpenSpec 狀態層），強大的工具鏈和自動化（Spec-Kit 工具層），以及深度協作和 Token 優化（BMAD 協作層 + Document Sharding）。整合四者能提供最完整的企業級解決方案，滿足審計、協作、品質、效率等多方面需求。 |
| **快速原型開發** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | Spec-Kit 或 BMAD (Quick Flow) | 原型開發的重點是速度。Spec-Kit 的 CLI 工具可以立即上手，支援 14+ AI 代理的無縫切換。BMAD 的 Quick Flow Track 提供輕量化流程（僅 Tech-Spec），適合快速迭代。OpenSpec 的變更管理在原型階段不是必要的，AISDLC 的完整流程會拖慢迭代速度。 |
| **需嚴格審計的專案** | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | AISDLC 2.0 (強化 AISDLC + OpenSpec) | 嚴格審計的專案需要完整的文檔追蹤鏈（PRD→FRD→SRD→AT）、強制反幻覺規則、多 Agent 交叉驗證，以及原子化的變更歷史記錄。AISDLC 的強制規則比 BMAD 的 C.O.R.E. 原則更適合審計需求。整合 AISDLC 追蹤鏈 + OpenSpec 變更管理 + Spec-Kit CI/CD 自動驗證，能提供最完整的審計證據鏈。 |
| **多角色協作專案** | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | BMAD + AISDLC | 多角色協作需要明確的角色定義、標準化工作流程、清晰產出標準和人機協作確認點。BMAD 的 19+ 專業化 Agent 和 Party Mode 提供深度協作機制，AISDLC 提供結構化流程和品質保證。兩者結合，既有靈活的即時協作又有嚴謹的流程規範。整合 Spec-Kit CLI 工具後可自動化協作流程，OpenSpec 的變更管理讓跨角色的變更審查更清晰。 |
| **開源專案貢獻** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | Spec-Kit + OpenSpec (+ BMAD 模組化) | 開源貢獻的關鍵是低門檻上手和清晰的變更提議。Spec-Kit 的 Dev Container 讓貢獻者 5 分鐘內完成環境設置，OpenSpec 的 changes 資料夾概念與 GitHub PR 流程天然契合。BMAD 的模組化設計和斜杠命令也友好，但整體學習曲線較陡。AISDLC 的完整角色和流程對開源貢獻者而言學習成本過高。 |
| **需求變更頻繁** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | BMAD + OpenSpec + AISDLC | 頻繁變更需要原子化的變更管理（OpenSpec changes 物件化）、靈活的流程適應（BMAD 規模自適應）和完整的影響分析（AISDLC 需求變更管理工作流程）。BMAD 的 Party Mode 可快速評估變更影響，規模自適應系統可根據變更大小調整流程深度。這個組合讓每個變更都獨立可審查、可回滾，同時保持敏捷性，避免變更衝突和遺漏。 |
| **API 開發專案** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | AISDLC 2.0 | API 開發需要強制的 API 規格文檔、完整的測試鏈（單元測試、集成測試、驗收測試）、清晰的版本管理。AISDLC 的 API 規格模板和測試追蹤鏈是核心優勢，BMAD 的 Document Sharding 對大型 API 文檔集合有效（節省 90%+ Token）。整合 OpenSpec 的變更管理可以清晰追蹤 API 版本演進，Spec-Kit 的 CI/CD 則能自動驗證 API 規格一致性。 |
| **大型複雜系統**<br>(40k+ tokens) | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | BMAD + AISDLC 2.0 | 大型複雜系統的核心挑戰是文檔量龐大導致的 Token 成本和處理效率問題。**BMAD 的 Document Sharding 技術**是關鍵優勢，可在實現階段節省 90%+ Token（只加載相關 Epic 分片）。19+ 專業化 Agent 系統適合處理多領域複雜性（前端、後端、資料庫、安全等）。整合 AISDLC 的文檔追蹤鏈確保複雜系統的可追溯性，OpenSpec 的變更管理處理頻繁的架構演進，Spec-Kit 提供自動化工具鏈。 |
| **創新型/探索性專案** | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ | BMAD | 創新型專案的特點是需求不明確、需要大量試錯和創意發想。**BMAD 的 C.O.R.E. 哲學**（通過引導性問題促使深度思考）和 **Party Mode**（多 Agent 即時頭腦風暴）最適合這類場景。規模自適應系統允許流程隨需求清晰度動態調整。AISDLC 的固定流程會限制探索空間，Spec-Kit 提供工具支援但缺少創意協作機制，OpenSpec 的變更管理在需求未穩定時價值有限。可選擇性整合 Spec-Kit 的 CLI 工具提升效率。 |
| **專家級團隊**<br>(深度客製化需求) | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | BMAD (+ 選擇性整合其他) | 專家級團隊希望 AI 作為助手而非流程執行者，需要高度客製化能力。**BMAD 的 Update-Safe Customization**（bmad/ vs bmd/ 分離）確保客製化在框架更新時受保護。19+ 專業化 Agent 可按需擴充，模組化設計（BMM, BMB, CIS）允許選擇性啟用。專家團隊可以基於 BMAD 靈活整合其他框架的特定功能：需要嚴謹性時整合 AISDLC 反幻覺規則，需要變更管理時整合 OpenSpec，需要自動化時整合 Spec-Kit。 |
