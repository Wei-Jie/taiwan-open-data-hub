# AISDLC 反幻覺檢查清單
# Anti-Hallucination Checklist

> **最終原則**：**寧可空白，不可捏造**

這份檢查清單確保所有使用 AISDLC framework 的用戶都遵守反幻覺規則，避免 AI 產生不實或未經驗證的內容。

---

## 📋 框架初始化檢查

### ✅ 載入階段檢查
- [ ] 是否已載入 `AISDLC_INIT.md`？
- [ ] 系統是否顯示「反幻覺規則已強制啟用」訊息？
- [ ] 是否看到「寧可空白，不可捏造」的核心原則提示？
- [ ] `anti-hallucination-rules.yaml` 是否被自動載入？

### ✅ Agent 配置檢查
- [ ] 載入的 Agent 是否在 core_principles 中包含反幻覺規則？
- [ ] Agent 配置中是否明確標示反幻覺規則為「最高優先級」？
- [ ] 是否驗證每個 Agent 都有專屬的反幻覺要求？

---

## 🛡️ 執行過程檢查

### ✅ 需求分析階段（SA Agent）
- [ ] 所有需求是否都能追溯到明確的用戶輸入或文檔？
- [ ] 是否避免自行創造用戶故事或需求？
- [ ] 每個需求項目是否都標註了來源？
- [ ] 遇到不確定的需求時是否立即詢問而非假設？
- [ ] 是否禁止使用「應該是」、「可能是」等模糊表述？

### ✅ 系統設計階段（SD Agent）
- [ ] 技術方案是否基於明確的需求文檔？
- [ ] 是否避免自行假設非功能性需求？
- [ ] 所有架構選擇是否有明確的技術依據？
- [ ] 是否禁止使用「通常做法」、「行業慣例」等模糊理由？
- [ ] 每個技術決策是否都說明了依據來源？

### ✅ 開發實現階段（Dev Agent）
- [ ] 代碼邏輯是否嚴格遵循 SRD/FRD 規格？
- [ ] 是否避免自行推斷未說明的業務邏輯？
- [ ] 邊界條件和錯誤處理是否基於明確的需求？
- [ ] 所有實現決策是否能追溯到規格文檔？
- [ ] 是否禁止自行補充未明確要求的功能？

### ✅ 測試驗證階段（QA Agent）
- [ ] 所有測試案例是否都能追溯到明確的需求？
- [ ] 是否避免基於推測設計測試？
- [ ] 測試場景是否都基於 FRD 中的驗收標準？
- [ ] 每個測試是否都標明了來源（FRD章節/用戶故事編號）？
- [ ] 是否禁止自行創造測試情境？

### ✅ 業務驗證階段（BA Agent）
- [ ] 所有業務理解是否都經過人類確認？
- [ ] 是否避免自行解釋業務邏輯？
- [ ] 利害關係人需求是否都有明確記錄？
- [ ] 所有關鍵業務決策是否都有人類確認記錄？
- [ ] 是否標註了確認時間和確認人？

### ✅ 產品決策階段（PM/PO Agent）
- [ ] 所有優先級判斷是否都有明確依據？
- [ ] 業務價值評估是否基於可驗證的資料？
- [ ] 產品決策是否都標明了依據來源？
- [ ] 是否避免基於主觀推測做決策？
- [ ] 決策是否透明且可追溯？

---

## 🔍 輸出品質檢查

### ✅ 內容完整性檢查
- [ ] 所有輸出內容是否都有明確來源？
- [ ] 是否避免包含任何未經確認的假設？
- [ ] 推論內容是否都明確標註為「推論」並說明依據？
- [ ] 不確定的項目是否都誠實聲明為「不確定」？
- [ ] 是否避免使用模糊語氣（「應該」、「可能」、「通常」）？

### ✅ 追溯性檢查
- [ ] 每個功能需求是否都能追溯到 PRD？
- [ ] 每個技術設計是否都能追溯到 FRD？
- [ ] 每個測試案例是否都能追溯到驗收標準？
- [ ] 每個實現決策是否都能追溯到 SRD？
- [ ] 需求追蹤鏈是否完整無斷裂？

### ✅ 文檔品質檢查
- [ ] 所有重要內容是否都標註了來源？
- [ ] 假設是否都明確記錄並標記為待確認？
- [ ] 不確定項目是否都明確聲明？
- [ ] 是否使用了標準的反幻覺回答格式？
- [ ] 文檔中是否避免了所有違規模式？

## ⚠️ 違規檢測

### 🚨 自動檢測模式（禁止出現）
檢查輸出中是否包含以下違規模式：

- [ ] **無**「應該是」
- [ ] **無**「可能是」
- [ ] **無**「我猜」
- [ ] **無**「通常來說」
- [ ] **無**「一般情況下」
- [ ] **無**「按照慣例」
- [ ] **無**「根據經驗」
- [ ] **無**「行業標準做法」

### 🚨 違規處理確認
- [ ] 發現違規時是否立即停止？
- [ ] 是否要求提供明確依據？
- [ ] 同類問題出現2次時是否標記為高風險？
- [ ] 同類問題出現3次時是否自動升級至專家？

## 🤝 人機協作檢查

### ✅ 強制確認點檢查
- [ ] 資訊不完整時是否立即停止並詢問？
- [ ] 存在多種理解可能性時是否尋求人類澄清？
- [ ] 規格解釋存在歧義時是否暫停並確認？
- [ ] 業務邏輯不明確時是否停止並詢問？
- [ ] 任何需要假設的情況是否都觸發了確認點？

### ✅ 零推測原則執行
- [ ] 是否絕對禁止「基於常識」自行決定？
- [ ] 是否絕對禁止「按照慣例」繼續工作？
- [ ] 是否絕對禁止任何形式的「合理推斷」？
- [ ] 30分鐘內未獲回應時是否啟動升級機制？
- [ ] 是否記錄了所有等待時間和升級過程？

### ✅ 確認品質驗證
- [ ] 人類回應是否完整回答了所有問題？
- [ ] 是否仍存在歧義或不確定點？
- [ ] 是否基於不完整回應進行了推測？（禁止）
- [ ] 連續3次回應不明確時是否觸發升級？

- [ ] 是否所有功能都能追溯到明確需求？
- [ ] 是否所有實現都符合設計規格？
- [ ] 發現不符時是否立即修正？

## 🎯 使用建議

### 初次使用時
1. 完整閱讀 `反幻覺prompt.md` 了解核心規則
2. 閱讀 `anti-hallucination-rules.yaml` 了解全局規則
3. 使用本檢查清單逐項確認
4. 首次執行 workflow 時特別留意所有檢查項目

### 日常使用時
1. 每次載入 AISDLC framework 時快速檢查「載入階段」
2. 每個 workflow 執行完畢後檢查「輸出品質」
3. 發現任何違規模式立即處理
4. 定期（每週）完整檢查一次全部項目

### 出現問題時
1. 立即停止當前工作
2. 使用本檢查清單定位問題
3. 查閱 `anti-hallucination-rules.yaml` 確認規則
4. 必要時尋求人類確認
5. 記錄問題和解決方案供未來參考

## ✅ 快速自檢問卷

在提交任何重要輸出前，快速回答以下問題：

1. **我的每一項陳述都有明確來源嗎？** （是/否）
2. **我是否做了任何未經確認的假設？** （是/否）
3. **我是否使用了任何模糊語氣？** （是/否）
4. **遇到不確定時我是否詢問了人類？** （是/否）
5. **我的輸出是否可以完全追溯到輸入？** （是/否）

**如果任何一個答案不符合反幻覺要求，請立即修正後再提交。**
