# 8. TDD 功能開發 (Test-Driven Development)

## 何時使用

當有單一、明確的功能需求（如一個 User Story 或 SRD 章節），且需要採用測試驅動開發（TDD）模式進行實作時使用。此流程強調規格的精確理解與 100% 合規，並內建 AI 幻覺防護機制，確保交付高品質、經過充分測試的程式碼。

**適用情境：**
- ✅ 實作單一、邊界清晰的功能
- ✅ 需求穩定且有完整的 SRD 或 User Story
- ✅ 專案要求高測試覆蓋率與規格合規性
- ❌ 不適用於多功能並行開發或大型模組重構

## 快速模板

```
我需要執行 TDD 開發流程來實作以下功能：

**功能描述**：
[在此貼上你的 User Story、SRD 描述或功能需求]

**技術規格**：
- **語言/框架**：[例如：Python/Django]
- **測試工具**：[例如：pytest]

**需求**：
請執行標準的 TDD 開發流程（紅-綠-重構），確保所有實作 100% 符合提供的規格，並交付完整的測試套件與功能程式碼。
```

## 完整範例

```
請執行增強版 TDD 開發流程，並啟用 AI 幻覺防護機制，實作以下功能：

**功能規格**
- **User Story**: 作為網站管理員，我希望能有一個 API 端點，可以根據用戶 ID 將其標記為「VIP」。
- **Acceptance Criteria**:
  1. API 端點為 `PUT /api/v1/users/{user_id}/vip`。
  2. 請求成功時，用戶的 `is_vip` 狀態應更新為 `true`。
  3. 如果用戶不存在，應返回 404 錯誤。
  4. 請求者必須是管理員角色，否則返回 403 錯誤。
  5. 成功時返回 200 OK 及更新後的用戶資料。

**技術需求**
- **框架**: Python FastAPI
- **測試框架**: pytest
- **資料模型**: User(id: int, username: str, is_vip: bool, role: str)

**品質要求**
- **測試覆蓋率**: 最低 95%。
- **AI 防護**: 啟用零推測原則，任何規格不明確處必須向我詢問確認。
- **規格合規**: 啟用即時規格合規驗證，確保開發過程不偏離需求。

**需求**：
請執行包含規格驗證、TDD 循環、深度合規驗證的完整流程，並管理好測試檔案的生命週期。
```

## 產出說明

此 workflow 產出經過嚴格驗證且品質可靠的功能模組。

### 您將獲得：

-   **測試套件**
    -   一組完整的測試案例檔案
    -   覆蓋所有成功路徑（Happy Path）
    -   覆蓋所有邊界條件（Boundary Cases）
    -   覆蓋所有異常情況（Error Cases）
    -   測試案例清晰命名，易於理解與維護

-   **實作程式碼**
    -   完全通過上述測試套件的功能程式碼
    -   遵循 SOLID 原則與最佳實踐
    -   包含適當的註解與文檔字串
    -   程式碼風格符合團隊規範

-   **規格合規報告**
    -   一份驗證報告，確認程式碼實作 100% 符合初始規格
    -   逐條檢查每個 Acceptance Criteria
    -   標示測試覆蓋率與通過狀態
    -   記錄規格理解過程中的確認點

-   **測試檔案管理記錄**
    -   測試檔案的創建、使用與清理日誌
    -   保持專案目錄整潔
    -   提供測試執行指令與環境要求

### TDD 流程說明

此 workflow 遵循經典的 **Red-Green-Refactor** 循環：

1.  **紅（Red）**: 先寫一個會失敗的測試
    -   根據 Acceptance Criteria 編寫測試案例
    -   執行測試，確認測試失敗（因為功能尚未實作）

2.  **綠（Green）**: 寫最少的程式碼讓測試通過
    -   實作功能，使測試通過
    -   不追求完美，先求可用

3.  **重構（Refactor）**: 優化程式碼
    -   在測試保護下，改善程式碼結構
    -   移除重複、提升可讀性
    -   確保測試仍然通過

### AI 幻覺防護機制

-   **零推測原則**: 遇到規格不明確處，必須向用戶確認
-   **30分鐘硬超時**: 單一任務超過30分鐘自動暫停並報告
-   **多 Agent 交叉驗證**: 由不同角色的 Agent 互相檢查
-   **即時規格合規驗證**: 開發過程中持續檢查是否偏離規格

### 互動指導

#### 預期 AI 確認問題

AI 系統會在以下時機詢問確認：
1. **規格理解確認**: 確保對需求的理解正確
2. **測試案例設計確認**: 確認測試案例覆蓋所有需求
3. **實作方向確認**: 確認技術實現方案
4. **規格偏差檢測時**: 發現實作與規格不符時

#### 人類參與要點

-   **及時回應**: 確保開發流程順暢
-   **提供明確答案**: 避免模糊不清的回應
-   **確認品質**: 仔細檢查最終交付物
-   **學習過程**: 觀察 AI 的 TDD 流程，提升自身技能
