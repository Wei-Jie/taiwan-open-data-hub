# 2. 需求驗證與文件化 (Validation and Documentation)

## 何時使用

當您已經完成初步的需求提取，並需要將這些需求轉化為正式、結構化的文件（如功能需求文件 FRD 或產品需求文件 PRD）時，請使用此 workflow。它能幫助您驗證需求的清晰度、完整性和可行性，並生成標準化的文件。

## 快速模板

```
作為一個專業的系統分析師（SA），請將我提供的需求清單轉換成一份結構化的功能需求文件（FRD）。

**需求清單:**
---
[此處貼上您的功能性與非功能性需求，通常是 Workflow 1 的產出]
---

**任務:**
1.  **分析與分組:** 將需求按模組或功能進行分組。
2.  **撰寫用戶故事:** 為每個功能性需求撰寫標準格式的用戶故事（As a [角色], I want [功能], so that [價值]）。
3.  **定義驗收標準:** 為每個用戶故事提供清晰、可測試的驗收標準（Acceptance Criteria）。
4.  **整合非功能性需求:** 將相關的非功能性需求（如性能、安全）整合到對應的功能模組中。
5.  **生成 FRD 文件:** 按照標準的 FRD 結構（包含封面、簡介、功能詳述、非功能性需求等）生成文件。
```

## 完整範例

```
作為一個專業的系統分析師（SA），請將我提供的需求清單轉換成一份結構化的功能需求文件（FRD）。

**需求清單:**
---
- **功能性需求:**
  - FR-1: HR 能夠建立、編輯和刪除職缺。
  - FR-2: HR 能夠查看所有候選人列表，並按職缺篩選。
  - FR-3: 系統儀表板需要顯示即時統計數據，如進行中面試數、已完成數。
  - FR-4: HR 可以導出指定職缺的候選人報告為 CSV 格式。
- **非功能性需求:**
  - NFR-1: 系統頁面加載時間必須在 3 秒內。
  - NFR-2: 所有對候選人資料的修改操作都必須被記錄。
---

**任務:**
1.  **分析與分組:** 將需求按模組或功能進行分組。
2.  **撰寫用戶故事:** 為每個功能性需求撰寫標準格式的用戶故事。
3.  **定義驗收標準:** 為每個用戶故事提供清晰、可測試的驗收標準。
4.  **整合非功能性需求:** 將相關的非功能性需求整合到對應的功能模組中。
5.  **生成 FRD 文件:** 按照標準的 FRD 結構生成文件。
```

## 產出說明

您將會得到一份專業的功能需求文件（FRD），其結構如下：

-   **1. 文件資訊:**
    -   版本、日期、作者等。

-   **2. 簡介:**
    -   項目目標和範圍。

-   **3. 功能需求詳述:**
    -   **模組 1: 職缺管理**
        -   **用戶故事 1.1:** 作為 HR，我希望能建立、編輯和刪除職缺，以便管理公司的招聘崗位。
        -   **驗收標準:**
            -   [ ] 點擊"新增職缺"按鈕，應彈出表單。
            -   [ ] 填寫職位名稱、描述後，職缺應出現在列表中。
            -   [ ] 編輯後的資訊應被正確保存。
            -   [ ] 刪除職缺時，系統應有確認提示。
    -   **模組 2: 候選人管理**
        -   **用戶故事 2.1:** 作為 HR，我希望能查看和篩選候選人列表，以便快速找到合適的人選。
        -   **驗收標準:**
            -   [ ] 候選人列表應預設顯示所有候選人。
            -   [ ] 可根據職缺進行篩選。
            -   [ ] 導出功能應能生成包含正確數據的 CSV 文件。

-   **4. 非功能性需求:**
    -   **4.1 性能:**
        -   所有頁面平均加載時間應小於 3 秒。
    -   **4.2 安全性:**
        -   系統需記錄所有對候選人資料的 CRUD 操作日誌。

這份文件為開發和測試團隊提供了清晰、無歧義的行動指南。
