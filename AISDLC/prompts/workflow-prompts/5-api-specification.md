# 5. API 規格生成 (API Specification Generation)

## 何時使用

當系統需求 (SRD) 已定義 API，但缺乏詳細規格文件時使用。適用於首次建立、更新過時規格、或為新 API 產生文件。確保 API 設計與需求一致，並提供開發團隊清晰的實作指引。

**適用情境：**
- ✅ SRD 已完成，包含 API 設計章節
- ✅ API 規格缺失或過時
- ✅ 新增 API 端點
- ❌ SRD 未完成或需求不穩定

## 快速模板

```
作為一個專業的系統設計師（SD），請執行 API 規格生成 workflow，我的項目資料如下：

- **SRD 文檔位置**：[SRD 文檔路徑或內容]
- **FRD 文檔位置**：[FRD 文檔路徑或內容]
- **需求**：請檢查 SRD，為其中定義的所有 API 生成詳細規格文檔，並建立 API 索引文件。
```

## 完整範例

```
作為一個專業的系統設計師（SD），請執行 API 規格生成 workflow，為電商系統的用戶認證模組生成 API 規格。

**專案資訊**
- **專案名稱**：E-Commerce Platform
- **系統模組**：用戶認證 (UserAuthentication)

**SRD 核心內容**
系統定義了以下 API：
1.  **用戶註冊**: `POST /api/v1/auth/register`
2.  **用戶登入**: `POST /api/v1/auth/login`
3.  **用戶登出**: `POST /api/v1/auth/logout`

**FRD 核心 User Story**
- **US-001 (用戶註冊)**: 作為新顧客，我希望能用 email 和密碼註冊帳號。
  - **AC-001-1**: 提供有效資料時，成功創建帳戶。
  - **AC-001-2**: 密碼需符合安全規則。
  - **AC-001-3**: 郵箱不能重複。
- **US-002 (用戶登入)**: 作為已註冊用戶，我希望能登入系統。
  - **AC-002-1**: 使用正確憑證可成功登入。
  - **AC-002-2**: 登入成功後返回 access token。

**需求**
1.  為每個 API 生成獨立的詳細規格文檔。
2.  創建 `API_Index.md` 索引文件。
3.  建立與 SRD、FRD 的追蹤鏈。
4.  遵循 RESTful 設計規範。
```

## 產出說明

此 workflow 將產出結構化的 API 文檔，確保開發的標準化與一致性。

### 您將獲得：

-   **API 規格文件**: 每個 API 的獨立 `.md` 文件，包含：
    -   端點路徑與 HTTP 方法
    -   請求參數（路徑、查詢、請求體）
    -   回應格式（成功與錯誤情況）
    -   狀態碼與錯誤碼說明
    -   請求/回應範例
    -   業務規則與限制條件

-   **API 索引文件**: `API_Index.md`，提供：
    -   所有 API 的分類目錄
    -   快速導航連結
    -   追蹤鏈關聯表（API → User Story → SRD 章節）

-   **更新的 SRD**: 在 SRD 文檔中嵌入 API 規格的連結，建立雙向追蹤關係

-   **狀態報告**: 一份關於 API 文檔完整性的報告，標示已完成與待處理的 API

### 品質保證

-   所有 API 規格與 SRD 定義保持一致
-   追蹤鏈完整，可追溯至對應的 User Story
-   遵循統一的文檔格式與命名規範
-   包含豐富的範例，降低實作歧義
