# 1. 需求提取 (Requirements Extraction)

## 何時使用

當您需要從多種來源（如會議記錄、訪談、客戶文件、甚至螢幕截圖）中，系統性地識別、整理和結構化功能性與非功能性需求時，請使用此 workflow。它特別適用於專案初期，幫助您快速建立需求的基線。

## 快速模板

```
作為一個專業的業務分析師（BA），請基於我提供的所有上下文，提取並組織功能性和非功能性需求。

**上下文:**
---
[此處貼上您的原始資料，例如會議記錄、訪談文字、功能列表或圖片]
---

**任務:**
1.  **提取功能性需求:** 識別所有用戶可以執行的具體操作或系統必須提供的功能。
2.  **提取非功能性需求:** 識別所有關於系統性能、安全性、可靠性、使用者體驗等方面的品質要求。
3.  **格式化輸出:** 使用 Markdown 表格呈現結果，包含 ID、需求類型、用戶角色、需求描述和來源。
```

## 完整範例

```
作為一個專業的業務分析師（BA），請基於我提供的所有上下文，提取並組織功能性和非功能性需求。

**上下文:**
---
**會議記錄: AI 面試系統 v2.0**

- **與會者:** PM (小明), RD (小華), UX (小美)
- **日期:** 2023-10-26
- **主題:** 規劃下一代 AI 面試系統

**討論重點:**

1.  **小明 (PM):** 我們需要一個更強大的後台，讓 HR 可以輕鬆管理職缺和候選人。目前的系統在候選人多的時候會變得很慢，需要提升性能。
2.  **小華 (RD):** 沒問題，我們可以重新設計數據庫架構。另外，建議增加一個儀表板，顯示即時的面試統計數據，例如進行中的面試、已完成的數量等。
3.  **小美 (UX):** 用戶反饋說，希望可以自定義面試問題的難度。同時，系統應該在 3 秒內加載完畢，以提升體驗。
4.  **小明 (PM):** 同意。安全性也很重要，所有用戶操作都需要被記錄下來，以供未來審計。HR 應該能夠篩選和導出候選人報告。

**任務:**
1.  **提取功能性需求:** 識別所有用戶可以執行的具體操作或系統必須提供的功能。
2.  **提取非功能性需求:** 識別所有關於系統性能、安全性、可靠性、使用者體驗等方面的品質要求。
```

## 產出說明

您將會得到一份結構化的需求分析報告，以 Markdown 表格形式呈現。

-   **功能性需求 (Functional Requirements):**
    -   **FR-1:** HR 管理職缺
    -   **FR-2:** HR 管理候選人
    -   **FR-3:** 系統提供儀表板，顯示即時面試統計
    -   **FR-4:** 用戶可自定義面試問題難度
    -   **FR-5:** HR 可篩選候選人報告
    -   **FR-6:** HR 可導出候選人報告

-   **非功能性需求 (Non-Functional Requirements):**
    -   **NFR-1:** 提升系統性能，避免在高負載下變慢
    -   **NFR-2:** 系統加載時間應少於 3 秒
    -   **NFR-3:** 記錄所有用戶操作以供審計
    -   **NFR-4:** 確保系統安全性

這個產出可以直接作為後續開發、測試和驗證的基礎，確保所有人都對需求有共同的理解。
