# 6. 文件一致性檢查 (Document Consistency Check)

## 何時使用

在專案關鍵節點（如 Sprint 結束、重大更新後）使用，確保多份文件（PRD, FRD, SRD, API, AT）之間的資訊保持同步與準確。此流程旨在自動盤點、驗證追蹤鏈、檢查引用連結並校驗內容邏輯。

**適用情境：**
- ✅ Sprint 或開發階段結束後
- ✅ 重大需求變更或文檔更新後
- ✅ 準備進行品質稽核前
- ❌ 專案初期文檔尚未齊備時

## 快速模板

```
請對我的 AISDLC 專案進行文件一致性快速檢查。

- **專案路徑**：[專案根目錄路徑]
- **檢查重點**：
  - 追蹤鏈條完整性 (FRD-SRD-AT)
  - 跨文件引用正確性
  - 內容邏輯一致性
- **需求**：執行檢查並生成一份包含問題清單與修復建議的報告。
```

## 完整範例

```
我需要執行完整的 AISDLC 文件一致性檢查，專案資訊如下：

**專案路徑**：`/project/ecommerce-system`

**檢查範圍**：
- [x] PRD 文檔
- [x] FRD 文檔
- [x] SRD 文檔
- [x] API 規格文檔
- [x] AT 測試文檔

**重點關注**：
- 用戶認證模組的追蹤鏈條是否完整。
- 訂單處理流程在 FRD 與 SRD 中的描述是否一致。
- API 規格中的連結是否都有效。

**已知問題**：
- FRD 中的部分 User Story 編號可能與 AT 不匹配。
- API 規格可能有部分更新未同步到 SRD。

**需求**：
請執行完整的文件一致性檢查 workflow，識別所有不一致、連結失效、追蹤鏈斷裂的問題，並生成一份詳細的檢查報告，包含問題的嚴重程度評估與修復優先級建議。
```

## 產出說明

此 workflow 旨在提升文檔品質與可信度，降低因資訊不一致導致的開發風險。

### 您將獲得：

-   **一致性檢查報告**: 一份詳細報告，包含：

    -   **文檔完整性分析**
        -   檢查必要文件是否缺失（PRD, FRD, SRD, API 規格, AT）
        -   評估文檔的結構完整度

    -   **追蹤鏈驗證**
        -   識別斷裂或不匹配的追蹤關係
        -   例如：US-001 → AC-001-X → AT-001
        -   標示孤立的 User Story 或測試案例

    -   **引用連結檢查**
        -   列出所有失效的跨文件連結
        -   驗證相對路徑的正確性
        -   檢查錨點連結是否有效

    -   **內容邏輯衝突**
        -   指出不同文件中描述矛盾之處
        -   例如：FRD 說支援 3 種付款方式，SRD 只設計了 2 種
        -   檢查數據模型定義的一致性

    -   **修復建議**
        -   提供具體、可操作的修復步驟
        -   按嚴重程度分級（Critical, High, Medium, Low）
        -   建議修復優先順序

### 報告分級標準

-   **Critical（嚴重）**: 追蹤鏈完全斷裂，影響開發與測試
-   **High（高）**: 內容描述矛盾，可能導致實作錯誤
-   **Medium（中）**: 連結失效或引用不完整，影響文檔可讀性
-   **Low（低）**: 格式不統一或輕微不一致，建議改進

### 使用技巧

-   **提供完整路徑**：確保專案路徑準確無誤
-   **明確檢查範圍**：勾選需要檢查的文檔類型
-   **突出重點**：在"重點關注"中說明最關心的問題
-   **利用已知問題**：提供已知問題有助於更快定位
-   **積極參與確認**：在人機協作確認點提供明確回饋
